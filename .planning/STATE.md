### Key Decisions (carried from v1.0)

- **Supervisor-Broker Pattern:** Research confirms this is the only way to manage elevated tasks like WFP while maintaining user-level CLI (2026-04-04).
- **WFP as Primary Network Backend:** Moving away from temporary firewall rules for true kernel-level enforcement (2026-04-04).
- **Named Job Objects:** Chosen for agent lifecycle management to ensure atomic stop/list capabilities (2026-04-04).
- **SID-Based Filtering:** Prioritized over App-ID to ensure child processes inherit network restrictions (2026-04-04).
- **Double-Launch Strategy:** Used `DETACHED_PROCESS` to decouple the supervisor from the parent terminal (2026-04-04).
- **Restricted Tokens:** Used to apply the session-unique SID to the process tree (2026-04-04).
- **RFC 3161 Timestamping:** Upgraded from legacy /t to /tr + /td sha256 (2026-04-05).
- **WFP Startup Orphan Sweep:** Enumerates NONO_SUBLAYER_GUID filters and removes stale ones at startup (2026-04-05).
- **Machine MSI Owns EventLog Registration:** SYSTEM\CurrentControlSet\Services\EventLog\Application\nono-wfp-service (2026-04-05).
- **MSRV 1.77:** Bumped from 1.74 to align with windows-sys 0.59 (2026-04-05).
- **WaitNamedPipeW Readiness Probe:** run_detached_launch() uses WaitNamedPipeW(50ms) per iteration on Windows (2026-04-05).
- **Single SID Generation Point:** Session SID generated once at ExecConfig construction (2026-04-06).
- **Driver Gate Removed:** activate_policy_mode no longer checks for a kernel driver binary artifact (2026-04-06).

The supervised child (which runs under a `CreateRestrictedToken(..., WRITE_RESTRICTED, ..., 1, &session_sid, ...)` token per `crates/nono/src/supervisor/restricted_token.rs`) hit `ERROR_ACCESS_DENIED` on `CreateFileW(\\.\pipe\nono-nono-cap-<hex>-pipe-<hex>-<hex>, GENERIC_READ | GENERIC_WRITE)` against the capability pipe. Initial hypothesis (commit `3c68377`): add a `(A;;0x0012019F;;;<session_sid>)` ACE to the pipe DACL so the second-pass (restricted-SIDs) access check finds an ACE matching a SID in `TokenRestrictedSids`. A standalone reproducer harness — the new `crates/nono-cli/examples/pipe-repro.rs` (parent creates pipe + restricted token + spawns self as `--child` via `CreateProcessAsUserW`; child calls `CreateFileW`; mirrors production's `create_restricted_token_with_sid` verbatim; accepts `--sddl-template` with `{sid}` placeholder for rapid iteration) — was used to test 13 SDDL variants. Results: V1 (baseline, session-SID ACE only) FAIL; V2 (`GA` mnemonic on session-SID ACE) FAIL; V3 (explicit `FILE_ALL_ACCESS = 0x001F01FF` on session-SID ACE) FAIL (refutes any mask-too-narrow hypothesis); V5 (Everyone `WD` ACE alone, no session-SID) FAIL; V4 (session-SID + Everyone `WD`) PASS; V8 (Authenticated Users `AU` alone) FAIL; V9 (`AU` + session-SID) PASS; V10 (`BU` Built-in Users + session-SID) PASS; V11 (literal user-SID `S-1-5-21-...-1002` + session-SID) FAIL; V14 (logon SID `S-1-5-5-0-8459283` + session-SID) PASS; V15 (logon SID alone, no session-SID) FAIL. **Conclusion:** Windows 11 26200's second-pass DACL access check for a `WRITE_RESTRICTED` token requires the DACL to contain TWO ACEs simultaneously: (1) an ACE whose SID is in the token's `TokenRestrictedSids` (the session SID); AND (2) an ACE whose SID is a GROUP SID in the token's `TokenGroups` with `SE_GROUP_MANDATORY` attribute. The existing `(A;;GA;;;OW)` baseline ACE does NOT satisfy (2) — `OW` is the Owner Rights virtual SID `S-1-3-4`, which is resolved against the object owner at access-check time and is apparently NOT considered equivalent to the user's mandatory group SIDs during the restricted-SID access check. The literal user-SID `S-1-5-21-...-1002` also does NOT satisfy (2) (V11 FAIL), so "user SID alone" is not enough; the ACE's SID must be one of the user's enabled group SIDs. **Fix (commit `938887f`):** `build_capability_pipe_sddl` in `crates/nono/src/supervisor/socket_windows.rs` now appends TWO ACEs when a per-session SID is provided: (1) `(A;;0x0012019F;;;<session_sid>)` — the necessary-but-insufficient restricting-SID ACE from `3c68377`; (2) `(A;;0x0012019F;;;<logon_sid>)` where the logon SID is retrieved at runtime via the NEW `current_logon_sid()` helper which calls `GetTokenInformation(GetCurrentProcess(), TokenGroups, ...)` and scans for the entry with `attributes & SE_GROUP_LOGON_ID != 0`, then formats the resulting `S-1-5-5-X-Y` triple via `ConvertSidToStringSidW`. The logon SID is unique per interactive-logon session (narrower than `WD` Everyone or `AU` Authenticated Users on the machine) — only processes in the SAME logon session have this SID in their token. Logon SID lookup failure is fail-closed (returns `NonoError::SandboxInit`). **Scope:** Windows-only.
