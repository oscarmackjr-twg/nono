## Accumulated Context

### Key Decisions (carried from v1.0)

- **Supervisor-Broker Pattern:** Research confirms this is the only way to manage elevated tasks like WFP while maintaining user-level CLI (2026-04-04).
- **WFP as Primary Network Backend:** Moving away from temporary firewall rules for true kernel-level enforcement (2026-04-04).
- **Named Job Objects:** Chosen for agent lifecycle management to ensure atomic stop/list capabilities (2026-04-04).
- **SID-Based Filtering:** Prioritized over App-ID to ensure child processes inherit network restrictions (2026-04-04).
- **Double-Launch Strategy:** Used `DETACHED_PROCESS` to decouple the supervisor from the parent terminal (2026-04-04).
- **Restricted Tokens:** Used to apply the session-unique SID to the process tree (2026-04-04).
- **RFC 3161 Timestamping:** Upgraded from legacy /t to /tr + /td sha256 (2026-04-05).
- **WFP Startup Orphan Sweep:** Enumerates NONO_SUBLAYER_GUID filters and removes stale ones at startup (2026-04-05).
- **Machine MSI Owns EventLog Registration:** SYSTEM\CurrentControlSet\Services\EventLog\Application\nono-wfp-service (2026-04-05).
- **MSRV 1.77:** Bumped from 1.74 to align with windows-sys 0.59 (2026-04-05).
- **WaitNamedPipeW Readiness Probe:** run_detached_launch() uses WaitNamedPipeW(50ms) per iteration on Windows (2026-04-05).
- **Single SID Generation Point:** Session SID generated once at ExecConfig construction (2026-04-06).
- **Driver Gate Removed:** activate_policy_mode no longer checks for a kernel driver binary artifact (2026-04-06).

A standalone reproducer harness — the new `crates/nono-cli/examples/pipe-repro.rs` (parent creates pipe + restricted token + spawns self as `--child` via `CreateProcessAsUserW`; child calls `CreateFileW`; mirrors production's `create_restricted_token_with_sid` verbatim; accepts `--sddl-template` with `{sid}` placeholder for rapid iteration) — was used to test 13 SDDL variants. Results: V1 (baseline, session-SID ACE only) FAIL; V2 (`GA` mnemonic on session-SID ACE) FAIL; V3 (explicit `FILE_ALL_ACCESS = 0x001F01FF` on session-SID ACE) FAIL (refutes any mask-too-narrow hypothesis); V5 (Everyone `WD` ACE alone, no session-SID) FAIL; V4 (session-SID + Everyone `WD`) PASS; V8 (Authenticated Users `AU` alone) FAIL; V9 (`AU` + session-SID) PASS; V10 (`BU` Built-in Users + session-SID) PASS; V11 (literal user-SID `S-1-5-21-...-1002` + session-SID) FAIL; V14 (logon SID `S-1-5-5-0-8459283` + session-SID) PASS; V15 (logon SID alone, no session-SID) FAIL. **Conclusion:** Windows 11 26200's second-pass DACL access check for a `WRITE_RESTRICTED` token requires the DACL to contain TWO ACEs simultaneously: (1) an ACE whose SID is in the token's `TokenRestrictedSids` (the session SID); AND (2) an ACE whose SID is a GROUP SID in the token's `TokenGroups` with `SE_GROUP_MANDATORY` attribute. The existing `(A;;GA;;;OW)` baseline ACE does NOT satisfy (2) — `OW` is the Owner Rights virtual SID `S-1-3-4`, which is resolved against the object owner at access-check time and is apparently NOT considered equivalent to the user's mandatory group SIDs during the restricted-SID access check. The literal user-SID `S-1-5-21-...-1002` also does NOT satisfy (2) (V11 FAIL), so "user SID alone" is not enough; the ACE's SID must be one of the user's enabled group SIDs. **Fix (commit `938887f`):** `build_capability_pipe_sddl` in `crates/nono/src/supervisor/socket_windows.rs` now appends TWO ACEs when a per-session SID is provided: (1) `(A;;0x0012019F;;;<session_sid>)` — the necessary-but-insufficient restricting-SID ACE from `3c68377`; (2) `(A;;0x0012019F;;;<logon_sid>)` where the logon SID is retrieved at runtime via the NEW `current_logon_sid()` helper which calls `GetTokenInformation(GetCurrentProcess(), TokenGroups, ...)` and scans for the entry with `attributes & SE_GROUP_LOGON_ID != 0`, then formats the resulting `S-1-5-5-X-Y` triple via `ConvertSidToStringSidW`. The logon SID is unique per interactive-logon session (narrower than `WD` Everyone or `AU` Authenticated Users on the machine) — only processes in the SAME logon session have this SID in their token. Logon SID lookup failure is fail-closed (returns `NonoError::SandboxInit`). **Scope:** Windows-only. D-21 preserved (zero diff on Linux/macOS files). **Verification:** `cargo test --release --lib -p nono supervisor::socket::tests::capability_pipe_admits_restricted_token_child_with_session_sid` PASS; live `nono run --profile claude-code -- aipc-demo.exe` advances past the pipe-connect and brokers 4 of 5 AIPC handle types end-to-end (the 5th, JobObject, hits a separate unrelated bug tracked as G-03 in 18-HUMAN-UAT.md). **MSDN note:** Microsoft's documentation for `CreateRestrictedToken` with `WRITE_RESTRICTED` describes the restricting-SID behavior but does NOT describe this group-SID co-requirement — the requirement was discovered by systematic SDDL iteration in the repro harness, not from reading documentation. Empirical-only finding; carried forward as a comment in `build_capability_pipe_sddl` so future maintainers understand why both ACEs are required. Debug session full trail in `.planning/debug/resolved/supervisor-pipe-access-denied.md` (2026-04-20).
- **WSAStartup gap in Phase 18 Plan 18-04 child SDK `reconstruct_socket_from_blob` (fix lands 2026-04-20):** After the supervisor-pipe WRITE_RESTRICTED fix (`938887f`) unblocked the broker pipeline end-to-end, the live `aipc-demo.exe` Socket broker request hit a latent bug in the Phase 18 Plan 18-04 child SDK surface: `crates/nono/src/supervisor/aipc_sdk.rs::reconstruct_socket_from_blob` called `WSASocketW(AF_UNSPEC, FROM_PROTOCOL_INFO, FROM_PROTOCOL_INFO, &proto_info, 0, WSA_FLAG_OVERLAPPED)` WITHOUT first calling `WSAStartup(MAKEWORD(2,2), ...)`. Result: `WSASocketW` returned `INVALID_SOCKET` and `WSAGetLastError()` returned `WSANOTINITIALISED = 10093`. Why unit tests passed: the Plan 18-04 smoke tests explicitly call `WSAStartup` themselves as part of their setup (since they construct the `WSAPROTOCOL_INFOW` blob via a real `WSADuplicateSocketW` in the test process first), so the WinSock subsystem was already initialized when `reconstruct_socket_from_blob` was called. The production-child path does NOT call `WSAStartup` anywhere — the child SDK is the first code in the child to touch WinSock. **Fix (commit `e4c1bfa`):** `reconstruct_socket_from_blob` now calls an internal `ensure_winsock_initialized()` helper BEFORE `WSASocketW`. The helper uses `static WINSOCK_INIT: OnceLock<Result<(), NonoError>> = OnceLock::new();` to ensure `WSAStartup(MAKEWORD(2, 2), &mut wsa_data)` is called exactly once per process; on subsequent calls the `OnceLock` returns the cached result. `WSACleanup` is intentionally NOT called — the child process exits cleanly and the OS releases WinSock state at process teardown; calling `WSACleanup` while other in-flight sockets exist would be a use-after-free footgun. Error handling: if `WSAStartup` returns non-zero, `ensure_winsock_initialized` returns `NonoError::SandboxInit("WSAStartup failed: <code>")`, propagated up through `reconstruct_socket_from_blob`, which propagates via `?` to the caller. **Scope:** Windows-only (`#[cfg(target_os = "windows")]` on the entire helper + its caller). D-21 preserved. **Verification:** live `aipc-demo.exe` Socket broker request now succeeds end-to-end; the child receives a valid reconstructed socket and subsequent `send`/`recv` operations work. **Why missed in planning:** Plan 18-04 RESEARCH did not flag `WSAStartup` as a precondition for `WSASocketW(FROM_PROTOCOL_INFO)` — MSDN's `WSASocketW` page mentions `WSAStartup` as a general WinSock prerequisite, but the `FROM_PROTOCOL_INFO` branch doesn't call it out specifically. The plan's test design also happened to coincidentally initialize WinSock via the test-side `WSADuplicateSocketW` call, masking the bug in the unit-test suite. **Followup guard:** Consider adding an integration test that exercises `reconstruct_socket_from_blob` in isolation (no preceding `WSADuplicateSocketW` in the test process) to catch similar uninitialized-subsystem bugs in future. Tracked in `18-HUMAN-UAT.md` G-06 adjacent (profile widening) but distinct. (2026-04-20)
