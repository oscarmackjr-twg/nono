## Implementation Decisions

### Area A — CLI surface & semantics

- **D-36.5-A1: `nono profile init --draft <name>` is the canonical create path.** Reinterpret REQ-PORT-CLOSURE-03 acceptance #1 literal text (`nono profile draft <name>`) as `nono profile init --draft <name>` for upstream byte-for-byte parity. Add a `pub draft: bool` field to `ProfileInitArgs` (cli.rs:1230). When `args.draft` is true, `cmd_init` writes to `profile-drafts/<name>.json` instead of `profiles/<name>.json`; all other init logic (name validation, `--extends`, `--groups`, `--description`, `--full`, `--force`, atomic write) reuses unchanged. No new `ProfileCommands::Draft` variant. REQ acceptance #1 is satisfied by the `init --draft` invocation.

- **D-36.5-A2: `validate --draft` and `validate --strict` are independent, composable flags.** Add `pub draft: bool` to `ProfileValidateArgs`. When `args.draft` is true, `cmd_validate` resolves the input from `profile-drafts/` instead of `profiles/`. The `--strict` flag (landed in Plan 36-01a — fails closed on legacy `override_deny` keys via `LegacyPolicyPatch` + `DeprecationCounter`) remains orthogonal: `validate --draft --strict <name>` is a valid combination that validates a draft *and* fails closed on legacy keys. Mirrors upstream.

- **D-36.5-A3: `nono profile promote <name>` uses interactive diff + `--yes` non-interactive flag.** New `ProfileCommands::Promote(ProfilePromoteArgs)` variant. `ProfilePromoteArgs { name: String, yes: bool }`. Default behavior:
  1. Resolve `profile-drafts/<name>.json` (error if missing).
  2. Resolve current canonical `profiles/<name>.json` (or detect missing → first-time promote, skip diff).
  3. Compute base-hash of current canonical profile bytes; compare against draft's recorded base-hash; on mismatch → `NonoError::ActionRequired` (D-36.5-B2).
  4. Run shadow-detection (D-36.5-D1); refuse-always on shadow.
  5. Render structural diff via `cmd_diff` rendering helper (D-36.5-D4).
  6. Prompt `[y/N]` unless `--yes` is set; on `n` or default → abort with `eprintln!` notice; on `y` or `--yes` → atomic rename of `profile-drafts/<name>.json` → `profiles/<name>.json`.

- **D-36.5-A4: `promote` accepts file-on-disk drafts only; no stdin path.** Atomicity + base-hash verification are well-defined only against an on-disk artifact. Stdin path deferred to v2.5-FU if CI use case surfaces.

### Area B — Storage path, error variant, hashing

- **D-36.5-B1: `profile-drafts/` lives under `nono_config_dir()` cross-platform** — `%APPDATA%\nono\profile-drafts\` on Windows, `~/.config/nono/profile-drafts/` on Linux/macOS. Reuse the existing `nono_config_dir()` helper at `crates/nono-cli/src/package.rs:270` (which resolves through `profile::resolve_user_config_dir()` at `crates/nono-cli/src/profile/mod.rs:2630`). Add `pub fn profile_drafts_dir() -> Result<PathBuf>` in `package.rs` mirroring the existing `profiles_dir()` (line 286) and `package_store_dir()` (line 274) shape. **Reinterprets REQ-PORT-CLOSURE-03 acceptance #4 literal `%LOCALAPPDATA%` as `%APPDATA%`** because the fork's actual PKG-02 implementation uses APPDATA (Plan 22-03 docs are stale — separate docs cleanup, not a 36.5 blocker). Rationale: sibling layout with `profiles/` and `packages/`, single helper, single mental model, single test surface.

- **D-36.5-B2: `NonoError::ActionRequired { expected: String, actual: String, resolve_via: String }` typed-struct variant.** Add to the `NonoError` enum at `crates/nono/src/error.rs`. `#[error("Action required: base-hash mismatch (expected: {expected}; actual: {actual}; resolve via: {resolve_via})")]` Display attribute matches REQ-PORT-CLOSURE-03 acceptance #3 wording with `resolve_via: nono profile init --draft --refresh <name>` (note: NOT `nono profile draft --refresh` per REQ literal — adjusted to match D-36.5-A1's `init --draft` subcommand shape). Display impl renders the operator-facing message; pattern-matching consumers see structured fields. Reused for shadow-refusal errors with a different `resolve_via` string (D-36.5-D3).

- **D-36.5-B3: C FFI maps `ActionRequired` to existing `NonoErrorCode::ErrConfigParse` (-9).** New arm in `bindings/c/src/lib.rs`: `NonoError::ActionRequired { .. } => NonoErrorCode::ErrConfigParse`. No new error code value in `bindings/c/src/types.rs:154`. Upstream-faithful per deferred-items.md § P34-DEFER-04b-2 line 74. C-side consumers can pattern-match on the Display string if structured distinction is ever needed; ABI stays stable.

- **D-36.5-B4: SHA-256 base-hash via existing `sha2` workspace dep.** Computes the hash over the byte content of the canonical profile JSON (post-serialization with stable key ordering). Matches Phase 22-03 PKG-04 signed-artifact verification, Phase 32 sigstore TUF root hashing, and Phase 04 undo ObjectStore content-addressable storage. One algorithm across the codebase. NOT Blake3 (not currently in dep tree; unnecessary speed advantage for sub-MB profile JSON).

### Area D — Shadowing safeguards & promote review UX

- **D-36.5-D1: Shadow-detection via existing helpers + embedded built-in lookup.** Detection order in `cmd_promote`:
  1. **Built-in shadow check** — name appears in the embedded built-in profile table loaded from `config/embedded.rs` (the same source `policy.rs` reads for built-in profile data; landed in Plan 36-01d). Read-only lookup.
  2. **Package-managed shadow check** — `crate::profile::is_package_managed_profile(name)` (existing helper at `profile/mod.rs:1932`-area; checks whether a profile at `profiles/<name>.json` is owned by a package via the lockfile + package symlinks).
  3. **User profile (no shadow conflict)** — replace allowed via atomic rename.
  Fail-closed on (1) or (2) → `NonoError::ActionRequired` (D-36.5-D3).

- **D-36.5-D2: NO `--force` override on `promote`.** Built-in and package-managed profiles are immutable from `promote` — full stop. **Deliberate security-posture deviation from upstream**, which has a `--force` flag. Rationale: a compromised draft must NOT be able to silently replace `claude-code` (or another widely-used built-in) by adding `--force`; the fork's threat model prioritizes structural impossibility over upstream parity here. Users wanting to override a built-in can still:
  - `nono profile init --force <name>` creates a *user* profile that shadows the built-in via the existing resolver precedence (profile/mod.rs:1985 — user profiles win over built-ins). This is a deliberate, single-step action with no draft-based indirection.
  - `nono package remove <ref>` removes a package-managed profile first, then `nono profile promote <name>` lands the draft normally.
  Document the deviation in `docs/cli/features/profiles-groups.mdx` profile-drafts section.

- **D-36.5-D3: Shadow-refusal error reuses `NonoError::ActionRequired` with a shadow-specific `resolve_via`.** Message shape:
  ```
  Action required: profile '<name>' is a <built-in|package-managed> profile and cannot be promoted over.
    Resolve via: rename the draft (`nono profile init --draft <new-name>`)
                 or override at the user layer (`nono profile init --force <name>` then edit)
                 or [if package-managed] remove the source package first (`nono package remove <ref>`)
  ```
  `expected` field holds the conflicting profile path; `actual` field holds the draft path; `resolve_via` holds the multi-line resolution string. Reusing `ActionRequired` (vs a new `ProfileShadowed` variant) keeps the C FFI surface narrow — one `ErrConfigParse` arm covers base-hash mismatch and shadow-refuse uniformly. Display impl distinguishes the two via the `resolve_via` content.

- **D-36.5-D4: Interactive diff renders structural JSON-aware diff via `cmd_diff` helper.** Call into the existing `profile_cmd::cmd_diff` (or its rendering helper at `crates/nono-cli/src/profile_cmd.rs` — find via `grep -n "cmd_diff\\|fn render_diff" crates/nono-cli/src/profile_cmd.rs` during research). Show added/removed/changed groups, capability changes, filesystem-deny/bypass_protection delta, command-policy delta. Consistent with `nono profile diff` UX. Sample output rendering:
  ```
  $ nono profile promote myagent
    Diff against /home/user/.config/nono/profiles/myagent.json:
      + security.groups: web-dev-read
      - filesystem.deny: ~/.ssh
      + filesystem.bypass_protection: ~/.config/git
    Promote draft? [y/N]
  ```
