## Objective Recap

Close two gaps from the 2026-04-20 Phase 18 HUMAN-UAT re-run on `nono run --profile claude-code -- aipc-demo.exe`:

- **G-03** — JobObject broker called `OpenJobObjectW("Local\\nono-aipc-<id>-aipc-demo-job")` and failed with `os error 2` (`ERROR_FILE_NOT_FOUND`) because the demo did not pre-create the named JobObject. Event/Mutex/Pipe/Socket brokers are CREATE-if-not-exists (via `CreateEventW`/`CreateMutexW`/`CreateNamedPipeW`/supervisor-owned `WSASocketW`); the JobObject broker was the odd one out. Per CONTEXT D-04/D-05 the design contract is "supervisor creates the named kernel object on demand."

- **G-04** — After G-03 failed, the supervisor dispatcher returned `ApprovalDecision::Approved` with `grant=None` — a protocol violation. The wire contract is `Approved ⟹ grant is Some, Denied ⟹ grant is None`. The child SDK's `send_capability_request` demultiplexer surfaced "supervisor granted but returned no ResourceGrant" as a hard error. Root cause: the dispatcher's `match result { Ok(g) => g, Err(e) => { tracing::warn!(...); None } }` silently dropped the error and left `decision` as Granted.

Both bugs lived in `crates/nono-cli/src/exec_strategy_windows/supervisor.rs` and shared the dispatcher's flow-control surface, so they were scoped into a single plan per the waves analysis.

**GREEN phase (commit `3493dd8`):** dispatcher flow-control rewrite in `handle_windows_supervisor_message` at lines ~1875-1950:

BEFORE:
```rust
let grant = if decision.is_granted() {
    let result: Result<Option<ResourceGrant>> = match request.kind { ... };
    match result {
        Ok(g) => g,
        Err(e) => {
            tracing::warn!("AIPC broker failure for kind {:?}: {}", request.kind, e);
            None   // BUG: decision still Granted, grant is None — protocol violation
        }
    }
} else {
    None
};
```

AFTER:
```rust
let (decision, grant) = if decision.is_granted() {
    let result: Result<Option<ResourceGrant>> = match request.kind { ... };
    match result {
        Ok(g) => (decision, g),
        Err(e) => {
            tracing::warn!("AIPC broker failure for kind {:?}: {}", request.kind, e);
            // G-04: flip decision so audit + wire both record Denied.
            let denied = nono::ApprovalDecision::Denied {
                reason: format!("broker failed: {e}"),
            };
            (denied, None)
        }
    }
} else {
    (decision, None)
};
```

Key structural change: the outer `let` now binds a tuple `(decision, grant)` instead of just `grant`. Rust's shadowing lets the tuple's `decision` slot re-bind the outer `decision` variable, so the subsequent `audit_log.push(audit_entry_with_redacted_token(&request, &decision, ...))` and `sock.send_response(&SupervisorResponse::Decision { request_id, decision, grant })` both see the flipped value. The `(Approved, grant=None)` shape is structurally unreachable from within `handle_windows_supervisor_message`.

Narrative comments in the two pre-existing containment-Job tests (`handle_denies_job_object_brokering_of_containment_job` at line ~3078 and its `_even_with_profile_widening` variant at line ~3218) refreshed to document that post-G-04 the wire decision is also Denied (the test assertions — `grant.is_none()` — are byte-identical and still pass because the broker helper still returns Err(containment-guard)).
