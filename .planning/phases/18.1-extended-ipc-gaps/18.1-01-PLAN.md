<objective>
Close G-02 by rerouting `TerminalApproval::request_capability` through the
D-04-locked `format_capability_prompt` helper. Phase 18 Plan 18-01 Task 4
landed the helper with full unit-test coverage for all 6 HandleKinds, but
IN-01 in `18-REVIEW.md` flagged that the live dispatcher never calls it —
it builds its own 3-line generic `Path / Access / Reason` block inline
(`terminal_approval.rs:41-49`). The 2026-04-20 UAT re-run on
`nono run --profile claude-code -- aipc-demo.exe` confirmed this: all 4
live Event/Mutex/Pipe/Socket prompts rendered the generic block instead
of the per-kind template. The fix is a single-site reroute inside
`TerminalApproval::request_capability` plus removal of the
`#[allow(dead_code)]` attributes on the five per-kind format helpers.

Purpose: Deliver the D-04-locked prompt UX end-to-end so approvers see
per-kind context (`proto=tcp host=example.com port=8080 role=connect`
instead of `Access: read-only`).

Output:

  inserted at the top of `request_capability`, legacy File block
  preserved as the `target.is_none()` fallback.

  `format_event_access`, `format_mutex_access`, `format_pipe_direction`,

- 5 regression tests added (one per new HandleKind: Event, Mutex, Pipe,

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sandboxed child → supervisor | CapabilityRequest arrives on the capability pipe carrying untrusted `target` fields (name, host, reason). `sanitize_for_terminal` is the enforcement point before any byte is emitted to the terminal. |
| supervisor → approver (terminal) | The rendered prompt text crosses from supervisor process to the approver's visual channel. ANSI escape injection via untrusted fields would let a malicious child steer the approver's eyes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18.1-01-01 | Spoofing | `format_capability_prompt` untrusted-field embedding | mitigate | `sanitize_for_terminal` is called on every untrusted field inside `format_capability_prompt` (name, host, reason) BEFORE embedding. Phase 18 Plan 18-01 Task 4 already implemented this; Phase 18.1 only routes the dispatcher through the helper. Regression-tested by `prompt_sanitizes_untrusted_target_strings` + `prompt_sanitizes_socket_host_string` (already in the test suite). |
| T-18.1-01-02 | Tampering | Mismatched (kind, target) shape | accept | `format_capability_prompt` has a defense-in-depth `_` arm that emits a clear "kind/target mismatch" placeholder rather than panicking. Phase 18 Plan 18-03 Task 3 verified via `prompt_renders_kind_target_mismatch_safely` test. Covered by Phase 18 — not a new surface in 18.1. |
| T-18.1-01-03 | Information disclosure | Reason field leaks sensitive data from child | accept | Same Phase 11 posture: the reason field is user-visible by design; the child is untrusted so any sensitive data it chooses to embed is its own operational mistake. `sanitize_for_terminal` only strips control bytes, not arbitrary text. |
