**Gathered:** 2026-04-21

## Phase Boundary

Close the 5 concrete gaps surfaced during the 2026-04-20 re-run of the Phase 18 HUMAN-UAT cookbook after the supervisor-pipe debug-session fix chain (`3c68377` session-SID ACE + `938887f` logon-SID ACE + `e4c1bfa` WSAStartup companion fix) unblocked the end-to-end `nono run --profile claude-code -- aipc-demo.exe` flow. G-01 (supervisor-pipe access denied) is RESOLVED and out of scope here — it is documented in `.planning/debug/resolved/supervisor-pipe-access-denied.md` for posterity.

**In scope — the 5 live gaps:**

1. **G-02** — CONIN$ approval prompts render generic `Path: ...` / `Access: read-only` / `Reason: demo` fields instead of the D-04-locked per-kind templates (`proto=tcp host=... port=... role=connect` for socket; `direction=read` for pipe; etc.). Dispatcher is wiring `format!` strings inline at the dispatch site rather than routing through `format_capability_prompt`.
2. **G-03** — JobObject broker calls `OpenJobObjectW("Local\nono-aipc-<session_id>-aipc-demo-job")` and fails with `os error 2` (`ERROR_FILE_NOT_FOUND`). The Event/Mutex/Pipe/Socket brokers CREATE-if-not-exists; the JobObject broker appears to only OPEN, breaking parity.
3. **G-04** — The supervisor returns `ApprovalDecision::Approved` with an empty `ResourceGrant` after an internal broker failure (G-03 triggered the observable case on JobObject, but the shape likely exists latent in all 5 broker handlers). The wire-protocol contract is: `Approved` MUST carry a valid `ResourceGrant`; `Denied` MUST carry a reason. Current code returns `Approved { grant: None }` when `OpenJobObjectW`/`CreateEventW`/`WSADuplicateSocketW` etc. fails AFTER user approval — a protocol violation that forces the child SDK to surface `"supervisor granted but returned no ResourceGrant"`.
4. **G-05** — WR-01's reject-BEFORE-prompt vs reject-AFTER-prompt invariant (Event/Mutex/JobObject reject BEFORE prompt via mask-validation gate; Pipe/Socket reject AFTER prompt pending a future fix) is NOT empirically verified. The 2026-04-20 UAT only exercised approval paths; no scripted deny-stage scenarios were run.
5. **G-06** — `capabilities.aipc` profile widening is not live-tested end-to-end. Plan 18-03 Deferred Issue #1 seeded `WindowsSupervisorRuntime.resolved_aipc_allowlist` with `AipcResolvedAllowlist::default()` pending a future plan that threads `Profile` through the dispatcher initialization. UNION semantic is unit-tested in `profile::tests`; end-to-end widening via a real `nono run --profile <widened>` invocation is not.

**Out of scope (intentionally):**

- **G-01 / supervisor-pipe access-denied** — RESOLVED on 2026-04-20 via the 3-commit fix chain. Reference material only (see `.planning/debug/resolved/supervisor-pipe-access-denied.md`).
- **WR-02 CompareObjectHandles EDR empirical test** — Deferred to v3.0 when an EDR-instrumented test runner is available (HUMAN-UAT Test 4 result line 64).
- **Sibling-to-sibling handle brokering** — Listed in REQUIREMENTS.md `Out of Scope (v2.1)`; existing supervisor-mediated flow is what's being fixed.
- **Linux/macOS handle brokering** — AIPC-01 is Windows-only per REQUIREMENTS.md line 295; D-21 Windows-invariance preserved.
- **New handle types beyond the 5 AIPC-01 kinds** — AIPC-01 closes the 5-kind contract; extensions are a separate phase.
- **Wire-protocol breaking changes other than the `Approved`-carries-`ResourceGrant` tightening in G-04.** The single well-motivated type change (Option<ResourceGrant> → ResourceGrant on the Approved variant, OR merging ApprovalDecision + ResourceGrant into a single enum where Approved carries the grant) is permitted; anything else (wire format, message envelopes, discriminator ordering) stays byte-identical.

## Implementation Decisions

### G-02: Route CONIN$ prompts through `format_capability_prompt`

- **D-01:** **Replace inline `format!` strings at the dispatcher prompt-construction site with calls to `format_capability_prompt(kind, target, access_mask, reason)`.** Per Plan 18-02 SUMMARY + 18-REVIEW IN-01, the helper exists in `crates/nono-cli/src/terminal_approval.rs` with per-kind branches for all 6 HandleKinds (File/Socket/Pipe/JobObject/Event/Mutex) and is covered by unit tests; the dispatcher in `crates/nono-cli/src/exec_strategy_windows/supervisor.rs` just doesn't call it — it emits its own 3-line generic prompt. The fix is a single-site replacement: find the site (likely in one of `handle_event_request` / `handle_mutex_request` / `handle_pipe_request` / `handle_socket_request` / `handle_job_object_request`, or a shared `show_approval_prompt` helper above them) and swap the inline `format!` for the D-04-locked template emitted by `format_capability_prompt`. Remove any `#[allow(dead_code)]` attribute on `format_capability_prompt` and per-kind `format_*` helpers once they have a live caller.

- **D-02:** **Delete the inline `format!` strings — do NOT keep them as fallback.** There is no scenario where the dispatcher should emit a prompt without going through `format_capability_prompt`; the helper already handles every HandleKind the dispatcher can receive. Keeping a fallback would reintroduce the drift that caused G-02 in the first place.

- **D-03:** **Add an end-to-end test that asserts the prompt text.** The existing integration tests exercise approval decisions (approved vs denied) but don't capture the rendered prompt. Add a test seam: either (a) a test-only `ApprovalBackend` trait impl that records the prompt text it was handed (preferred — no real CONIN$), or (b) a stdin/stderr-capturing test harness that drives the real terminal path. Option (a) is strongly preferred because it runs on every CI push, not just on a Windows host with a TTY. The assertion checks that the rendered text contains the per-kind field labels (`proto=`, `host=`, `port=`, `role=` for Socket; `direction=` for Pipe; `access=` for Event/Mutex/JobObject; `path=` for File).

### G-03: JobObject broker CREATE-vs-OPEN semantic

- **D-04:** **Align JobObject broker semantic with Event/Mutex/Pipe/Socket: CREATE-if-not-exists.** Per the HUMAN-UAT gap writeup, the design-intended contract across all 5 broker kinds is "supervisor creates the named kernel object on demand, brokers a handle into the child." Event/Mutex brokers use `CreateEventW`/`CreateMutexW` which atomically create-or-open. Pipe broker uses `CreateNamedPipeW` (create) + `bind_aipc_pipe`. Socket broker uses `WSADuplicateSocketW` on a supervisor-opened socket. JobObject broker uses `OpenJobObjectW` which only OPENS. The fix: replace `OpenJobObjectW` with `CreateJobObjectW(name)` which atomically creates-or-opens (semantically matches `CreateEventW` behavior). `GetLastError` returns `ERROR_ALREADY_EXISTS` if the object pre-existed — that's fine, we still got a handle. Apply the D-05 default-mask allowlist check on the resulting handle.

- **D-05:** **No demo changes required.** The alternative (make the demo pre-create the JobObject before requesting brokering) was rejected per the HUMAN-UAT writeup's "supervisor creates the named kernel object on demand" design principle and the REQUIREMENTS.md AIPC-01 contract which says the supervisor brokers the handle. The demo stays as-is; the broker gets the semantic-aligning fix.

- **D-06:** **Preserve the containment_job runtime guard.** The existing `CompareObjectHandles` check at `exec_strategy_windows/supervisor.rs:~1678` that refuses to broker the supervisor's own containment Job regardless of profile widening MUST remain byte-identical. The D-04 change is in the broker-handle-acquisition step ONLY — the subsequent guard + mask-validation + approval-prompt pipeline is unchanged.

- **D-07:** **Add an integration test.** Extend `crates/nono-cli/tests/aipc_handle_brokering_integration.rs` with a JobObject round-trip case that exercises the full CREATE-on-demand → prompt → DuplicateHandle → child-SDK-receives path. The existing 5-test suite (per Plan 18-03 SUMMARY) had a JobObject case with `BrokerTargetProcess::current()` — it must already be passing on the unit-level but not exercising the end-to-end broker flow through the dispatcher. Either extend the existing case or add a new one alongside it.

### G-04: Broker-failure propagation + wire-protocol tightening

- **D-08:** **Fix broker handlers to flip `ApprovalDecision::Approved` → `Denied { reason: "broker failed: <error>" }` when the internal kernel-object operation fails post-approval.** The bug lives in all 5 handler functions (`handle_event_request`, `handle_mutex_request`, `handle_pipe_request`, `handle_socket_request`, `handle_job_object_request` — per HUMAN-UAT G-04: "The Event/Mutex/Pipe/Socket handlers likely have the same latent shape but never hit it because their CREATE-if-not-exists semantics don't fail"). After the handler calls its `broker_*_to_process` backend and the backend returns an error, the outer function currently does something like `ApprovalDecision::Approved { grant: None }` or silently drops the error and returns `Approved` without a grant. The fix: propagate the backend error through a proper `NonoError` or `Result<ResourceGrant, String>` boundary and map error-case to `Denied { reason: format!("broker failed: {}", err) }`. Reason string is verbose-on-purpose for debuggability (same rationale as D-07 in Phase 18's CONTEXT.md); not a security-sensitive surface.

- **D-09:** **Tighten the wire-protocol `Approved` shape so empty `ResourceGrant` becomes a compile-time error.** The current shape (per the HUMAN-UAT G-04 "consider tightening the wire-protocol types" suggestion) likely has `ApprovalDecision::Approved` + a sibling `grant: Option<ResourceGrant>` field, OR the handler returns `ApprovalDecision` and `Option<ResourceGrant>` separately. Refactor to: `ApprovalDecision::Approved(ResourceGrant)` (Approved carries the grant inline, non-optional) OR a single `enum BrokerOutcome { Approved { grant: ResourceGrant }, Denied { reason: String } }` that replaces the split. The compile-time enforcement means the "approved with empty grant" bug cannot reoccur — the compiler refuses to build code that tries to return Approved without a concrete ResourceGrant. If this refactor would bloat the diff (touches every broker handler + child SDK demultiplexer + test suite), scope it to ONLY the supervisor → child direction of the wire (the `SupervisorResponse::Decision` shape) — the request direction is unchanged.

- **D-10:** **Add regression tests that inject a failure at the broker step and assert the response is `Denied { reason: ... }`, not `Approved { grant: None }`.** Mechanism: introduce a test seam on each of the 5 broker backends (`broker_event_to_process` etc.) that a test harness can substitute with a failing impl. Simplest implementation: a test-only `#[cfg(test)]` constant or once-cell that a test can flip to force the next call to return `Err(...)`. Alternative: make each `broker_*_to_process` signature take a `&dyn BrokerBackend` parameter in tests; may be too invasive. Favor the `#[cfg(test)]` flip-flag approach for minimum diff, or a per-handler `MockBrokerBackend` trait only inside the test module. Each of the 5 handlers gets one such failure-injection test; the test asserts `matches!(response, ApprovalDecision::Denied { reason, .. } if reason.contains("broker failed"))`.

- **D-11:** **The child SDK `send_capability_request` demultiplexer gets a matching simplification.** Once `ApprovalDecision::Approved` carries `ResourceGrant` inline, the child SDK's "supervisor granted but returned no ResourceGrant" branch becomes unreachable — delete the branch. This tightens the SDK surface to match the wire: Approved implies a grant, Denied implies a reason, nothing in between.

### G-05: WR-01 reject-stage invariant verification

- **D-12:** **Add 4 scripted deny-stage integration tests in `crates/nono-cli/tests/aipc_handle_brokering_integration.rs` (or a new sibling file if the existing one becomes too dense):**
  1. **Event with mask outside DEFAULT_MASK** → expect `Denied { reason }` WITHOUT the approval backend being invoked (mask-validation gate fires BEFORE prompt, per Plan 18-01 D-07 + summary invariant).
  2. **Mutex with mask outside DEFAULT_MASK** → same as Event (reject BEFORE prompt).
  3. **JobObject with mask outside DEFAULT_MASK and no profile widening** → same (reject BEFORE prompt; the existing `handle_denies_job_object_with_terminate_mask_no_profile_widening` test already covers the "denied" half but does NOT assert prompt was skipped — extend it).
  4. **Pipe with `direction=read+write` under default profile** → expect `Denied` AFTER approval backend was invoked (reject AFTER prompt — documented WR-01 current behavior pending future fix). Assertion via backend-invocation count.
  5. **Socket with privileged port (`port=80`)** → expect `Denied` for WR-01 reasons; empirically verify whether this rejects BEFORE or AFTER prompt (HUMAN-UAT G-05 scenario #4 — the privileged-port check is unconditional and lives at `supervisor.rs:1461`; need to determine from the code whether the check is placed before or after the approval-backend call and document the actual stage).

- **D-13:** **Test-seam mechanism: a `ProbeApprovalBackend` impl that records calls.** Same mechanism as G-02 D-03 — a test-only impl of the approval-backend trait/signature that records whether it was invoked, with what arguments. The deny-stage tests assert: "backend was invoked 0 times" for reject-BEFORE-prompt cases; "backend was invoked exactly 1 time" for reject-AFTER-prompt cases. A single shared test fixture across G-02 D-03 and G-05 D-13 is the preferred path — one mock, two test suites.

- **D-14:** **Document the WR-01 verdict in the test file's module docstring.** After the 5 tests land and the actual stages are empirically verified, add a module-level comment that states: "As of Phase 18.1, Pipe ReadWrite and Socket privileged-port rejections happen AFTER the approval prompt (WR-01). This is documented current behavior; a future fix to route these through the same mask-validation gate as Event/Mutex/JobObject would eliminate the UX inconsistency." The verdict is NOT to fix WR-01 in Phase 18.1 — that's a separate product decision (HUMAN-UAT Test 2 result notes "a product decision is required on whether to accept the deviation or schedule a follow-up"). Phase 18.1 only verifies the current behavior.

### G-06: Profile widening end-to-end wiring

- **D-15:** **Thread `Profile` through the `WindowsSupervisorRuntime` initialization so `resolved_aipc_allowlist` is sourced from the loaded profile, not `AipcResolvedAllowlist::default()`.** The supervisor-runtime construction site (likely in `crates/nono-cli/src/exec_strategy_windows/supervisor.rs` or `supervised_runtime.rs`) currently seeds the field with `AipcResolvedAllowlist::default()` per Plan 18-03 Deferred Issue #1. The call site has access to the resolved `Profile` object (it has to — that's what drives the whole sandbox configuration). Thread the profile into the construction: `WindowsSupervisorRuntime::new(..., profile.resolve_aipc_allowlist())` instead of `...default()`. The `resolve_aipc_allowlist(&self) -> AipcResolvedAllowlist` method already exists on `Profile` (per Plan 18-03 SUMMARY + verified at `profile/mod.rs:706`) and returns the UNION of (hard-coded defaults, profile override).

- **D-16:** **Add an end-to-end integration test that exercises the widened path.** Construct a test Profile in-memory (via `Profile::from_json` or the existing test-construction helper) with `capabilities.aipc.pipe: ["read+write"]` and verify: (a) a `request_pipe` with `direction=read+write` gets through the mask-validation gate AND reaches the approval prompt; (b) the same request under the default profile (no widening) rejects at the mask-validation gate without reaching the prompt. Use the `ProbeApprovalBackend` from D-13 to count prompt invocations. Socket variant: widen `capabilities.aipc.socket: ["bind", "listen"]` and verify a `request_socket` with `role=bind` reaches the prompt; same request under default profile rejects BEFORE prompt.

- **D-17:** **Revert test: remove the widening and re-run, assert default behavior returns.** Per HUMAN-UAT G-06 Next Action #3, the test must verify "removing the widening from the profile and re-running returns to default behavior (no stale caching)." Easiest implementation: the test constructs TWO profiles (widened + default), runs the same request against each, asserts different outcomes. No runtime "profile reload" mechanism needed — two fresh `WindowsSupervisorRuntime` instances, one per profile.

- **D-18:** **Plan 18-03's `AipcResolvedAllowlist` + `AipcConfig` + `CapabilitiesConfig` types stay unchanged.** The wiring change is ONE LINE at the supervisor-runtime construction site. The types and their serde/validation logic are correct (per Plan 18-03 SUMMARY: 9 profile tests PASS + capability_handler_tests 23/23 PASS); they just aren't plumbed end-to-end. D-18 is a non-change decision: do NOT refactor `AipcResolvedAllowlist` into something different.

## Specific Ideas

### G-02 — exact prompt text to assert in tests (copy from Phase 18 CONTEXT D-04):

- Event: `[nono] Grant event access? name=<n> access=<wait|signal|both> reason="<r>" [y/N]`
- Mutex: `[nono] Grant mutex access? name=<n> access=<wait|release|both> reason="<r>" [y/N]`
- Pipe: `[nono] Grant pipe access? name=<n> direction=<read|write|read+write> reason="<r>" [y/N]`
- Socket: `[nono] Grant socket access? proto=<tcp|udp> host=<h> port=<p> role=<connect|bind|listen> reason="<r>" [y/N]`
- JobObject: `[nono] Grant Job Object access? name=<n> access=<query|set_quotas|terminate|...> reason="<r>" [y/N]`
- File: `[nono] Grant file access? path=<x> access=<read|write|read+write> reason="<r>" [y/N]`

Each test asserts the rendered text equals the expected string with concrete field values substituted (after `sanitize_for_terminal` scrubbing).

### G-03 — exact Win32 API swap:

- Current (in `broker_job_object_to_process`): `OpenJobObjectW(JOB_OBJECT_ALL_ACCESS, FALSE, LPCWSTR(name))` returning `ERROR_FILE_NOT_FOUND` when the name doesn't exist.
- Fix: `CreateJobObjectW(ptr::null(), LPCWSTR(name))` which create-if-not-exists, returns `ERROR_ALREADY_EXISTS` via `GetLastError()` when it opened an existing object. Apply the caller-validated access mask via subsequent `DuplicateHandle` with the mask; `CreateJobObjectW` returns a full-access handle, `DuplicateHandle` downgrades to the validated mask before handing to the child.

Alternative if `CreateJobObjectW` semantics don't match exactly (check MSDN for the create-or-open story): fall back to the Event broker's pattern (`CreateEventW` + `GetLastError() == ERROR_ALREADY_EXISTS` check).

### G-04 — the shape that should not exist:
