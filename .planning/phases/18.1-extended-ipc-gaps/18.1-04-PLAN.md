<objective>
Close G-05 by landing 5 scripted deny-stage integration tests that
empirically verify the WR-01 reject-stage invariant across all 5 new
AIPC HandleKinds. Per Plan 18-01 SUMMARY the invariant is:
Event/Mutex/JobObject reject BEFORE the approval prompt (pre-broker
mask-validation gate at supervisor.rs:1834-1866); Pipe/Socket reject
AFTER the approval prompt (the direction/role/port checks live INSIDE
their per-kind helpers). The 2026-04-20 HUMAN-UAT did not exercise any
deny paths — only approvals — so the invariant remained empirically
unverified. Phase 18.1 G-05 closes that gap without fixing WR-01 itself
(WR-01 fix is explicitly deferred to a future product decision per
18-HUMAN-UAT Test 2 + CONTEXT D-14).

Purpose: Lock the reject-stage semantic into regression tests so any
future refactor that accidentally moves a mask-validation check from
pre-broker to post-broker (or vice versa) breaks CI.

- 5 new tests in `capability_handler_tests` (inside
  `crates/nono-cli/src/exec_strategy_windows/supervisor.rs`), each named
  `wr01_<kind>_rejects_{before,after}_prompt_<condition>`.

  `handle_denies_job_object_with_terminate_mask_no_profile_widening`
  test (line 3025) — the "BEFORE prompt" stage is currently only

**NOTE on "mask outside DEFAULT_MASK" for Event/Mutex:** The executor
must read `crates/nono/src/supervisor/policy.rs` to identify a valid
"extra" bit to test with. If `policy.rs` exports only the default-mask
bits (no "outside" constants), the test can construct an arbitrary
out-of-allowlist mask by ORing in bits known to fail the subset check.
The simplest approach: use a value like `0x10000000` (a high bit that is
NOT in any default mask) as the "out-of-allowlist" mask. This does not
need to correspond to a real Win32 semantic — the point is that
`mask_is_allowed(kind, 0x10000000, default_mask)` returns false.
