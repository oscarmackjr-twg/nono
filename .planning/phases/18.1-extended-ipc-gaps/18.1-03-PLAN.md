<objective>
Close G-06 by threading the loaded `Profile`'s resolved AIPC allowlist
through the Windows supervisor construction path end-to-end. Plan 18-03
Deferred Issue #1 seeded `WindowsSupervisorRuntime.resolved_aipc_allowlist`
with `AipcResolvedAllowlist::default()` pending a future plan that wires
`Profile::resolve_aipc_allowlist()` through. That future plan is this one.

Purpose: The UNION semantic (hard-coded defaults ∪ profile widening) is
already unit-tested in `profile::tests` (9 tests PASS per Plan 18-03
SUMMARY); the missing piece is the one-line construction-site swap plus
the end-to-end verification that a `nono run --profile <widened>`
invocation actually sees the widened allowlist at request-dispatch time.

Output:
- `SupervisorConfig<'a>` (Windows branch) gains a new field
  `pub aipc_allowlist: AipcResolvedAllowlist`.
- `WindowsSupervisorRuntime::initialize` reads `supervisor.aipc_allowlist`
  into `self.resolved_aipc_allowlist` (replaces the `default()` seed).
- `SupervisedRuntimeContext<'a>` gains
  `pub(crate) loaded_profile: Option<&'a crate::profile::Profile>`.
- `execute_supervised_runtime` resolves `ctx.loaded_profile.map(|p| p.resolve_aipc_allowlist()).transpose()?.unwrap_or_default()`
  and passes it into `supervisor_cfg.aipc_allowlist`.
- `sandbox_prepare.rs` passes `loaded_profile` through to the execution
  runtime (it's already in `PreparedProfile`; the plumbing extends to
  `SupervisedRuntimeContext`).
- 3 new tests in `capability_handler_tests`:
  1. Pipe ReadWrite + widened allowlist → reaches backend (approval
     prompt), backend returns Granted, broker step fires.
  2. Pipe ReadWrite + DEFAULT allowlist → rejected BEFORE backend
     (pre-broker direction-validation, current Pipe behavior per
     18-CONTEXT WR-01 is AFTER prompt — but the ALLOWLIST-check portion
     is unambiguously pre-prompt for mask-based kinds).
     **Important for the pipe case:** Pipe's direction-check at
     `supervisor.rs:1410` lives INSIDE `handle_pipe_request` (post-approval
     per WR-01). G-06's Pipe test therefore needs to verify the backend
     WAS invoked but the helper `Err(..)`d → dispatcher flips to Denied
     (G-04 semantic). This is the correct current behavior.
  3. Socket Bind role + widened allowlist → reaches backend; same
     caveat (Socket role-check at supervisor.rs:1489 is also inside
     the per-kind helper, post-approval per WR-01).
- All pre-existing 20+ capability_handler_tests keep passing — test-site
  `&AipcResolvedAllowlist::default()` calls to
  `handle_windows_supervisor_message` are unchanged (the new field is
  only on the OUTER `SupervisorConfig`, not on the dispatcher function).
</objective>
