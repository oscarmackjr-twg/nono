<objective>
Close G-03 (JobObject broker uses OPEN-only `OpenJobObjectW` instead of
CREATE-if-not-exists `CreateJobObjectW`) and G-04 (dispatcher returns
`Approved` with `grant=None` when the per-kind broker helper fails
AFTER user approval — protocol violation). Both bugs live in
`crates/nono-cli/src/exec_strategy_windows/supervisor.rs` and share the
dispatcher's flow-control surface, so they land in a single plan.

Purpose: (a) Restore parity with Event/Mutex/Pipe/Socket CREATE-semantic
so JobObject brokering works end-to-end for `aipc-demo.exe`; (b) fix
the empty-grant protocol violation by flipping `Approved` → `Denied`
when the internal broker step errors. The wire-protocol shape
(`SupervisorResponse::Decision { decision, grant: Option<ResourceGrant> }`)
stays byte-identical — per CONTEXT D-09 the tighter refactor to
`Approved(ResourceGrant)` would cascade into Phase 18 Plan 18-04's
aipc_sdk demultiplexer and its 23 capability_handler_tests, exceeding
this plan's context budget. The enforcement instead lives at the
dispatcher flow-control boundary: `Err` from a per-kind broker helper
MUST flip `decision` to `Denied` before `send_response` is called, so
the (Approved, None) shape is structurally unreachable from within
this module.

Output:
- 1-line-ish swap inside `handle_job_object_request`:
  `OpenJobObjectW(policy::JOB_OBJECT_ALL_ACCESS, 0, wide.as_ptr())` →
  `CreateJobObjectW(std::ptr::null_mut(), wide.as_ptr())`.
  Preserves the containment-Job CompareObjectHandles guard and the
  subsequent DuplicateHandle MAP-DOWN step.
- ~10-line flow-control flip inside `handle_windows_supervisor_message`
  (currently lines 1875-1937): when `result: Result<...>` is `Err(e)`,
  overwrite `decision` with `Denied { reason: format!("broker failed: {e}") }`
  and set `grant = None` — before the audit_log push + send_response.
  The `tracing::warn!` stays as diagnostic context.
- 5 new tests in `capability_handler_tests` module (one per HandleKind:
  Event, Mutex, Pipe, Socket, JobObject) that inject a deterministic
  broker-step failure and assert the response is `Denied { reason }`
  containing `broker failed`.
- Existing 2 containment-Job tests (line 3078 and 3218) updated
  narrative comment only — they still pre-create via `CreateJobObjectW`
  and still assert the same guard fires (no functional change needed).
</objective>
