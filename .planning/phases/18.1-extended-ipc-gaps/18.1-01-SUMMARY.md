# Phase 18.1 Plan 18.1-01: G-02 Route CONIN$ Prompts Through D-04 Per-Kind Templates Summary

**Rerouted the live `TerminalApproval::request_capability` dispatcher through the D-04-locked `format_capability_prompt` helper via a new testable `build_prompt_text(&CapabilityRequest) -> String` wrapper; removed 5 `#[allow(dead_code)]` attributes on the per-kind format helpers that had been dead since Phase 18 Plan 18-01; added 6 regression tests capturing rendered prompt text for all 6 HandleKind shapes including the Phase 11 legacy File-kind byte-identical preservation.**

## Objective Recap

Close G-02 from the 2026-04-20 Phase 18 HUMAN-UAT re-run on `nono run --profile claude-code -- aipc-demo.exe`. Before this fix, `TerminalApproval::request_capability` emitted a generic 3-line `Path: / Access: / Reason:` block for every capability request regardless of `HandleKind`, so the live Event/Mutex/Pipe/Socket prompts rendered generic text instead of the D-04-locked per-kind templates (`proto=tcp host=... port=... role=connect` for socket, `direction=read` for pipe, `access=wait+signal` for event, etc.). The per-kind templates existed in `format_capability_prompt` with full unit-test coverage since Phase 18 Plan 18-01 Task 4 — but the dispatcher never called the helper. 18-REVIEW.md IN-01 flagged this, and Plan 18.1-01 was the dedicated single-site reroute fix.
