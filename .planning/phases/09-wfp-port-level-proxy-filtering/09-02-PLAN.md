In `crates/nono-cli/src/execution_runtime.rs`, add a pre-flight guard **BEFORE** the `start_proxy_runtime` call at line 165. The guard must fire BEFORE `start_proxy_runtime` runs because:

1. When `proxy.active == false`, `start_proxy_runtime` returns early WITHOUT mutating caps (proxy_runtime.rs:186-191). If caps were already set to `ProxyOnly { port: 0, ... }` by the profile path (capability_ext.rs:535), they remain in that state post-call — but a post-call guard that only checks `proxy_handle.is_none()` would not distinguish "legitimately no proxy requested" (caps = Blocked/AllowAll) from "proxy-only requested but no config" (caps = ProxyOnly).
2. Putting the check BEFORE the call captures user **intent** directly: caps in `ProxyOnly` mode + `proxy.active == false` = configuration error per D-02.

Insert the guard immediately before line 165 (`let active_proxy = start_proxy_runtime(proxy, &mut caps)?;`):

```rust
    // Fail-secure guard (D-02): if the user's intent is proxy-only mode
    // (caps were set to ProxyOnly by a profile or credential path at
    // capability_ext.rs:535) but `proxy.active` is false (no network profile,
    // no credentials, no upstream proxy — see proxy_runtime.rs lines 52-78),
    // the sandboxed process would have no proxy to route traffic through.
    // Fail before `start_proxy_runtime` and before any WFP/sandbox activation.
    if matches!(caps.network_mode(), nono::NetworkMode::ProxyOnly { .. }) && !proxy.active {
        return Err(nono::NonoError::SandboxInit(
            "Cannot use proxy-only mode without a network profile or credential configuration."
                .to_string(),
        ));
    }

    let active_proxy = start_proxy_runtime(proxy, &mut caps)?;
```

**Why this check is correct:**

- `proxy.active` is computed in `prepare_proxy_launch_options` (proxy_runtime.rs:52-78) BEFORE `start_proxy_runtime` is called. It reflects the user's configured intent, not post-mutation state.
- When the profile path sets `ProxyOnly { port: 0, ... }` but no credentials / domain / upstream proxy are configured, `proxy.active` evaluates to `false`.
- `start_proxy_runtime` explicitly does NOT mutate caps when `proxy.active == false` (proxy_runtime.rs:186-191), so a pre-call check sees the same `caps.network_mode()` a post-call check would see — but the pre-call check fires BEFORE any runtime setup, matching the CONTEXT.md D-02 requirement to fail "before touching WFP".
- Using `!proxy.active` (not `proxy_handle.is_none()`) makes the intent explicit and avoids relying on post-mutation signal.

**Defense-in-depth (optional, NOT in the primary fix):** If desired for future-regression protection, a secondary assertion after `start_proxy_runtime` could check `matches!(caps.network_mode(), ProxyOnly) && proxy_handle.is_none()`. This is OPTIONAL — the primary guard above already closes D-02. Do NOT add the secondary assertion unless it can be written without shadowing the primary guard's error message. The primary guard is sufficient.

Note: The existing `proxy_env_vars` injection path at lines 177-179 is already complete on Windows. No changes needed there. The `ExecConfig` at lines 260-269 already includes `env_vars` which contains the proxy env vars. This was verified in the research phase.
