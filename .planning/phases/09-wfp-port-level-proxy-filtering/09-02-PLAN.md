The guard checks: if the caps are now in `ProxyOnly` mode but the proxy handle is `None`, something went wrong and we must fail-secure.

Insert after line 167 (`let proxy_handle = active_proxy.handle;`):

```rust
    // Fail-secure guard: if caps are in ProxyOnly mode (proxy was supposed to start)
    // but no proxy handle was returned, the sandboxed process would have no proxy
    // to route traffic through. This is a configuration error per D-02.
    if matches!(caps.network_mode(), nono::NetworkMode::ProxyOnly { .. }) && proxy_handle.is_none()
    {
        return Err(nono::NonoError::SandboxInit(
            "Cannot use proxy-only mode without a network profile or credential configuration. \
             The proxy was expected to start but no proxy handle was created."
                .to_string(),
        ));
    }
```

This guard fires in the edge case where:
1. A profile sets `ProxyOnly` mode with `port: 0` placeholder (capability_ext.rs line 535)
2. `start_proxy_runtime` starts the proxy and mutates caps to ProxyOnly with real port
3. BUT somehow the handle is None (should not happen if proxy.active was true, but is a safety net)

The more likely scenario this catches: future code changes that could set ProxyOnly mode without starting a proxy. The guard is fail-secure by design.

Note: The existing `proxy_env_vars` injection path at lines 177-179 is already complete on Windows. No changes needed there. The `ExecConfig` at lines 260-269 already includes `env_vars` which contains the proxy env vars. This was verified in the research phase.
