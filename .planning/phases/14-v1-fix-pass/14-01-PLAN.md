<objective>
Fix Bug #3 residual — the detached-path `STATUS_DLL_INIT_FAILED (0xC0000142)` that still kills console grandchildren even after the `WRITE_RESTRICTED` fix for Bug #2 landed. The bug is the last blocker for `nono run --detached` on Windows and blocks four Phase 13 UAT items (P05-HV-1, P07-HV-3, P11-HV-1, P11-HV-3).

Purpose: Close Phase 14 roadmap success criterion #1 so the v1.0 Windows Parity milestone can be archived with every VERIFICATION.md honestly promoted to `passed`.

Output: A token-construction change in `restricted_token.rs` (primary direction: session SID as token *group*, not restricting SID) plus regression tests exercising detached-console-child initialization AND sandbox-boundary enforcement, with a documented decision checkpoint before falling back to alternative directions.

This plan is `autonomous: false` because Task 1.5 is a human-verify checkpoint: the fallback direction (if Direction 1 fails the Task 1 smoke) must be acknowledged by the developer before Task 2/3 proceed, because each fallback carries a different security trade-off.

<security_acceptance_gate>
**This gate is non-negotiable.** A fix that regresses any of the following is rejected, regardless of whether it makes the smoke test pass:

1. A child sandboxed under the new token path must NOT be able to read a filesystem path outside its `CapabilitySet` (the sandbox boundary is enforced by the Windows backend in `crates/nono/src/sandbox/windows.rs`, not by the token alone; verify it still holds).
2. The session SID must still be matchable by WFP `FWPM_CONDITION_ALE_USER_ID` — either as a restricting SID OR as a token group. WFP matches against any SID present in the token, so promoting session SID from restricting to group preserves this property. Document the exact property relied on in the fix's code comment. If direction 2 is taken (null token + AppID WFP), network identity moves to AppID — that trade-off must be acknowledged at Task 1.5 and documented in the commit body.
3. The fix must NOT fall back to the unrestricted supervisor token without an equivalent kernel-level boundary. If the chosen direction replaces SID-based WFP filtering with AppID-based WFP filtering (direction 2 fallback), the plan must verify the AppID filter is actually installed and active on the detached path's grandchild.
4. `--block-net` + the new token path must still result in all outbound connections from the sandboxed child being dropped by WFP. A network-negative regression smoke is in scope for this plan (Task 3 manual-smoke step).

If the primary direction (token group SID) cannot satisfy all four properties, fall back per the decision tree in Task 1 below and acknowledge the trade-off at the Task 1.5 checkpoint.
</security_acceptance_gate>
