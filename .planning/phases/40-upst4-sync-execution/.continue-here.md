## Root-cause hypothesis (errors 4, 5)

Plan 40-03's cherry-pick of upstream `6472011` (and possibly `78114e6`) introduces references to `nono::undo::SessionMetadata.rollback_status` field and `nono::undo::RollbackStatus` type in `audit_ledger.rs:383+435` and `audit_session.rs:396+`. These names compile on Windows but fail on non-Windows.

**Likely causes (investigate in order):**

1. **The field/type exists in upstream `nono::undo` but the fork's `nono::undo` has different cfg-gating.** Compare:
   - `crates/nono/src/undo/types.rs` (or wherever `SessionMetadata` is defined) — is `rollback_status: RollbackStatus` field present and unconditional? Or is it `#[cfg(target_os = "windows")]`?
   - `crates/nono/src/undo/mod.rs` — is `RollbackStatus` re-exported cross-platform?
2. **The cherry-pick depends on a Phase 32-or-later fork-only undo extension that's Windows-gated.** Search `git log -p crates/nono/src/undo/` for the introduction of `rollback_status` field.
3. **The cherry-pick should have added the field to the fork's `SessionMetadata` cross-platform but the executor only patched the Windows arm.** Check 40-03 commits `96886ae9` and `7831c47f` for any new `#[cfg(target_os = "windows")]` blocks the executor added defensively.

**Likely fixes (after diagnosis):**

- If field/type IS Windows-only-gated in fork's undo module: either un-gate it (cross-platform make it `Option<...>` or `RollbackStatus::None`-style default), or wrap the audit_ledger / audit_session references in `#[cfg(target_os = "windows")]` blocks.
- If the cherry-pick lost a re-export: add the missing `pub use undo::RollbackStatus;` (or similar) to `crates/nono/src/lib.rs`.
- Three lint errors (1/2/3) are CR-A class — same fix pattern as commit `a72736bb`: gate the `use` line on `#[cfg(target_os = "windows")]` (or `#[cfg(test)]` for error #3 since that's a test-only import).
