**IMPORTANT: `prepare_live_windows_launch` takes `config: &ExecConfig<'_>` (immutable borrow), so the SID CANNOT be populated there.** The primary Windows `ExecConfig` construction site is in `execution_runtime.rs` at line 283 (the `#[cfg(target_os = "windows")]` block), where `flags.session.session_id` is accessible. Populate `session_sid` there:
```rust

let config = exec_strategy::ExecConfig {
    command: &command,
    resolved_program: &resolved_program,
    caps: &caps,
    env_vars,
    cap_file: cap_file.as_deref(),
    current_dir: &current_dir,
    session_sid: Some(exec_strategy_windows::restricted_token::generate_session_sid()),
```

Note: `session_sid` is always `Some` on the Windows execution path because every sandboxed Windows launch needs a unique SID for WFP filter scoping. The `Option` type exists for test ergonomics and to support the App-ID fallback path (D-02) where callers may explicitly pass `None`. Check whether `restricted_token::generate_session_sid` is `pub(super)` and adjust visibility to `pub(crate)` if needed so `execution_runtime.rs` can call it, or alternatively call it through a re-export in the `exec_strategy_windows` module.

**Part C: Wire session_sid into build_wfp_target_activation_request (D-02) in network.rs**

1. Add `session_sid: Option<&str>` parameter to `build_wfp_target_activation_request`:
```rust
pub(super) fn build_wfp_target_activation_request(
    policy: &nono::WindowsNetworkPolicy,
    target_program: &Path,
    outbound_rule: &str,
    inbound_rule: &str,
    session_sid: Option<&str>,
) -> WfpRuntimeActivationRequest {
    let mut request = build_wfp_runtime_activation_request(policy);
    request.target_program_path = Some(target_program.display().to_string());
    request.outbound_rule_name = Some(outbound_rule.to_string());
    request.inbound_rule_name = Some(inbound_rule.to_string());
    request.session_sid = session_sid.map(str::to_string);
    request
}
```

1. Generates a session SID (line 806)
2. Builds a WFP activation request with wrong `request_kind = "activate_policy_mode"` (line 812)

4. Creates a restricted token (line 822)

This is a duplicate/conflicting activation path. WFP activation is already handled by `prepare_network_enforcement` -> `install_wfp_network_backend` which runs BEFORE `spawn_windows_child` is called. The fix:

**Remove the WFP activation portion but keep the restricted token creation**, using `config.session_sid` instead of generating a new SID:

Replace lines 805-827 (the `session_sid` generation through the token creation block) with:
```rust
    // Create restricted token if session SID was generated during network enforcement setup
    let h_token = if let Some(ref sid) = config.session_sid {
        restricted_token::create_restricted_token_with_sid(sid)?.h_token
    } else if should_use_low_integrity_windows_launch(config.caps) {
        create_low_integrity_primary_token()?.raw()
    } else {
        std::ptr::null_mut() // Use current process token (CreateProcessW)
    };
```

This means:
- No duplicate WFP activation request (the wrong `request_kind` code is deleted)
- No second call to `generate_session_sid()` (uses the SID already in `config.session_sid` from Task 1)
- Token creation still happens using the same SID that was used for WFP filter installation
- The `session_id` parameter to `spawn_windows_child` may no longer be needed for SID generation — audit whether it is used elsewhere in the function. If only used for the removed block, the parameter can stay for now (it is used in other logging/state).

Also remove the now-unused import of `build_wfp_runtime_activation_request` and `run_wfp_runtime_request` from launch.rs if they are no longer referenced in this file. Check with `cargo check`.
