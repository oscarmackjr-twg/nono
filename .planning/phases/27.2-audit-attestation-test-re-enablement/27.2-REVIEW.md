# Phase 27.2: Code Review Report

**Reviewed:** 2026-05-09T18:00:00Z
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 27.2 wires `cmd_verify` (and as a cascade `cmd_show`) through
`audit_session::load_session`, relocates the audit-attestation bundle and
audit-events ledger to `<audit_root>/<id>/`, and adds a sibling
`AuditState.rollback_dir` so `--rollback`'s SnapshotManager continues to
write `changes/`, `snapshots/`, and `session.json` under
`<rollback_root>/<id>/`. The integration tests are re-enabled with a local
`ScopedEnvVar` RAII guard, and tracing output is routed to stderr to keep
`--json` stdout pure.

The bundle-target relocation closes the non-repudiation hole that was the
phase's stated goal — but it also splits authoritative metadata across two
directories and the loader's two-root iteration does NOT survive
`load_session_metadata` failure. Empirical reproduction (Windows host,
`nono run --audit-integrity --rollback --allow <writable_dir>`): session.json
is written exclusively to `<rollback>/<id>/`, and `nono audit verify <id>`
returns `ERROR: Session not found: Failed to read session metadata
.../audit/<id>/session.json: The system cannot find the file specified.
(os error 2)`. The integration test `rollback_signed_session_verifies_from_audit_dir_bundle`
masks this by passing only `--allow-cwd`, which produces no User-source
write capability — `derive_tracked_paths` returns empty, so
`initialize_rollback_state` short-circuits to `Skipped`, no snapshot is
captured, the audit-only finalize branch runs, and session.json lands at
the audit dir. Real `--allow <dir>` users will hit BL-01.

A second BLOCKER (BL-02): the new `rollback_dir` derivation in
`create_audit_state` uses `nono_home_dir().join(".nono").join("rollbacks")`,
which on Windows production resolves to `%USERPROFILE%\.nono\rollbacks` and
diverges from `rollback_session::rollback_root()`'s
`%LOCALAPPDATA%\nono\rollbacks`. Sessions written by `nono run --rollback
--audit-integrity` will be invisible to `nono rollback list` /
`nono rollback restore` on Windows. (NONO_TEST_HOME masks this in tests
because both helpers route through the same override root.) The bug is
inherited from pre-Phase-27.2 code — the file is in scope and the new
rollback_dir field codifies the divergence.

The remaining findings are warnings: a `?` cascade in `load_session` that
defeats the audit-first/rollback-fallback design even when only
canonicalization fails; a parallel
`ensure_rollback_session_dir`/inlined-derivation pair that will drift; an
inert `#[cfg(target_os = "windows")]` arm in a `#[cfg(not(target_os =
"windows"))]`-gated unit test; an integration-test `ScopedEnvVar` that
duplicates the in-tree `EnvVarGuard` semantics without a process-wide env
lock; the deprecation-shim warning helper that is unreachable for the
common combo-session shape (gated behind BL-01); and a discoverability gap
where `read_capability_decisions_from_ledger` succeeds for audit-only
sessions but reads from the wrong directory for combo sessions when
session.json eventually loads.

## Blockers

### BL-01: `audit_session::load_session` returns hard error on combo sessions; `?` defeats audit-first/rollback-fallback design

**File:** `crates/nono-cli/src/audit_session.rs:207`
**Issue:** For sessions started with `--audit-integrity --rollback` AND a
non-empty derived tracked-paths set, `finalize_supervised_exit` writes
`session.json` exclusively to `<rollback>/<id>/` (via
`manager.save_session_metadata`, line 667 of `rollback_runtime.rs`). The
audit dir is created and populated with `audit-attestation.bundle` +
`audit-events.ndjson` only.

`load_session` then iterates `[audit_root, rollback_root]`:

```rust
for root in roots.into_iter().flatten() {
    let dir = root.join(session_id);
    if !dir.exists() { continue; }                          // audit/<id> EXISTS
    let canonical_root = root.canonicalize().map_err(...)?; // ok
    let canonical_dir = dir.canonicalize().map_err(...)?;   // ok
    if !canonical_dir.starts_with(&canonical_root) { continue; }
    let metadata = SnapshotManager::load_session_metadata(&dir)?;  // FAILS, ? propagates
    ...
}
```

When the audit dir exists but holds no `session.json`, the `?` on line 207
short-circuits the loop and the legacy rollback root is never consulted.
`nono audit verify <id>` returns `Session not found: Failed to read session
metadata .../audit/<id>/session.json: The system cannot find the file
specified. (os error 2)`. Empirically reproduced on Windows with `nono run
--audit-integrity --rollback --allow-cwd --allow <dir> -- cmd /c echo
nono-test`.

The integration test
`rollback_signed_session_verifies_from_audit_dir_bundle` only passes because
`--allow-cwd` does NOT register as a User-source `Write|ReadWrite` capability
(the gate in `derive_tracked_paths`, line 306-316), so
`initialize_rollback_state` returns `(None, Skipped)` and the audit-only
finalize branch (line 685-712) writes session.json to the audit dir. As soon
as a user adds an explicit `--allow <writable>` to a `--rollback
--audit-integrity` invocation, verification breaks.

This is the literal regression that Phase 27.2-01's plan was supposed to
prevent (REQ-AAHX-01: "audit-only sessions findable"). The phase closed it
for the audit-only case but introduced a parallel hole for combo sessions
because Phase 27.2-02 split the metadata target without teaching the loader
about the split.

**Fix:** Either (a) write session.json to BOTH audit and rollback dirs in
`finalize_supervised_exit` (closes the loader hole at the source), or (b)
turn the per-root metadata-load failure into a `continue` so the loop falls
through to the rollback root. (a) is the safer choice because audit-first
discovery callers (`audit list`, `audit show`, future tools) all assume
session.json lives at audit dir.

```rust
// In rollback_runtime.rs::finalize_supervised_exit, after line 667:
if let Some(audit_state) = audit_state {
    // Mirror session.json to the audit dir so audit_session::load_session
    // finds it even when snapshot artifacts live elsewhere.
    if audit_state.session_dir != manager.session_dir() {
        nono::undo::SnapshotManager::write_session_metadata(
            &audit_state.session_dir,
            &meta,
        )?;
    }
}
```

(Or change `load_session` line 207 from `?` to `match ... continue` and
ensure the `audit_event_count > 0` check at line 209 still excludes
rollback-only sessions correctly.)

### BL-02: `create_audit_state` rollback_dir derivation diverges from `rollback_session::rollback_root()` on Windows production

**File:** `crates/nono-cli/src/rollback_runtime.rs:240-241`
**Issue:** The `rollback_dir` fallback when no `--rollback-dest` is supplied:

```rust
None => {
    let home = crate::config::nono_home_dir()?;
    home.join(".nono").join("rollbacks")
}
```

uses `nono_home_dir()`, which on Windows production resolves to
`dirs::home_dir()` ≈ `%USERPROFILE%`. Result:
`%USERPROFILE%\.nono\rollbacks`.

But `rollback_session::rollback_root()` (the canonical helper used by
`discover_sessions`, `load_session`, `total_storage_bytes`,
`remove_session`) on Windows uses `crate::config::user_state_dir()` which
returns `%LOCALAPPDATA%\nono\rollbacks` (rollback_session.rs:30-37):

```rust
#[cfg(target_os = "windows")]
{
    crate::config::user_state_dir()
        .ok_or(NonoError::ConfigParse(...))
        .map(|root| root.join("rollbacks"))
}
```

These paths diverge on Windows production. Sessions written by
`finalize_supervised_exit` via `manager.save_session_metadata` (which uses
the path computed in `create_audit_state`) land at
`%USERPROFILE%\.nono\rollbacks\<id>\session.json`. `nono rollback list`,
`nono rollback restore`, `nono session ls` etc. read from
`%LOCALAPPDATA%\nono\rollbacks\` and will not see those sessions.

The bug is pre-existing — the lines were inherited verbatim from before
Phase 27.2 (the same `home.join(".nono").join("rollbacks")` derivation
existed in the previous `session_dir` arm). However, Phase 27.2 codified
the bug into a new `AuditState.rollback_dir` field that is now also the
canonical source for the snapshot SnapshotManager root in
`initialize_rollback_state` (line 504-507). The file is in review scope.
NONO_TEST_HOME masks the divergence in tests because `nono_home_dir()` and
`user_state_dir()` both honor it.

The companion helper `audit_session::ensure_rollback_session_dir` (lines
70-94) does it correctly — it calls `rollback_session::rollback_root()`
directly. So the codebase already has the right pattern; this call site
just doesn't use it.

**Fix:** Replace the inlined derivation with the canonical helper:

```rust
let rollback_dir = if rollback_active {
    let dir = crate::audit_session::ensure_rollback_session_dir(
        &session_id,
        rollback_destination,
    )?;
    Some(dir)
} else {
    None
};
```

This also removes the duplicated `create_dir_all` + `set_permissions(0o700)`
block (lines 245-260) that already exists in `ensure_rollback_session_dir`.
Note: `ensure_rollback_session_dir` is currently `#[allow(dead_code)]` —
this fix removes the attribute since it gains a live caller.

## Warnings

### WR-01: `load_session` propagates root-canonicalize failure instead of continuing to next root

**File:** `crates/nono-cli/src/audit_session.rs:193-199`
**Issue:** The `?` after `root.canonicalize()` returns the function early
when the audit root cannot be canonicalized (e.g., transient filesystem
permission error, symlink loop, antivirus interference on Windows):

```rust
let canonical_root = root.canonicalize().map_err(|e| {
    NonoError::SessionNotFound(format!(
        "Cannot canonicalize audit root {}: {}",
        root.display(), e
    ))
})?;
```

This defeats the dual-root design: a transient error on the primary audit
root prevents the legacy rollback root from being consulted, even though
the legacy root may be perfectly readable. The error message mentions
"audit root" specifically, which means this also leaks to the user as the
audit-root-specific failure rather than a generic SessionNotFound.

**Fix:** Treat root-canonicalize failure the same as
"directory-doesn't-exist" — log a debug line and `continue`:

```rust
let canonical_root = match root.canonicalize() {
    Ok(p) => p,
    Err(e) => {
        tracing::debug!(
            "Skipping audit root {} (cannot canonicalize): {}",
            root.display(), e
        );
        continue;
    }
};
```

### WR-02: `ensure_rollback_session_dir` and `create_audit_state`'s inlined derivation are parallel implementations destined to drift

**File:** `crates/nono-cli/src/audit_session.rs:69-94` and
`crates/nono-cli/src/rollback_runtime.rs:236-264`
**Issue:** Both functions create `<rollback_root>/<id>/`, set 0o700
permissions on Unix, and emit identical-shape error messages. The audit_session
helper uses the canonical `rollback_session::rollback_root()` (correct);
the rollback_runtime inlining uses `nono_home_dir().join(".nono").join("rollbacks")`
(see BL-02). When BL-02 is fixed by inlining the same divergent path,
fixing the divergence in only one place leaves the other broken. This is
the structural reason BL-02 exists.

**Fix:** Adopt the BL-02 fix above (call `ensure_rollback_session_dir` from
`create_audit_state`), drop the inlined derivation, and remove
`#[allow(dead_code)]` from `ensure_rollback_session_dir`. This collapses
the parallel implementations to one.

### WR-03: Unreachable `#[cfg(target_os = "windows")]` arm inside Unix-only test

**File:** `crates/nono-cli/src/audit_session.rs:366-375`
**Issue:** The test is gated:

```rust
#[cfg(not(target_os = "windows"))]
#[test]
fn discover_sessions_excludes_rollback_backed_entries() {
    let _env_lock = lock_env();
    let tmp = tempfile::tempdir().unwrap();
    let home = tmp.path().to_string_lossy().to_string();
    #[cfg(target_os = "windows")]
    let _env = EnvVarGuard::set_all(&[("USERPROFILE", &home)]);
    #[cfg(not(target_os = "windows"))]
    let _env = EnvVarGuard::set_all(&[("HOME", &home)]);
    ...
```

Because the outer `#[cfg(not(target_os = "windows"))]` already excludes
Windows compilation, the inner `#[cfg(target_os = "windows")]` arm is
dead. The conditional is harmless but misleading — it suggests Windows
support that the test doesn't actually have. The companion comment at
lines 360-365 says Windows is deferred to Plan 22-05b.

**Fix:** Remove the dead `#[cfg(target_os = "windows")]` arm and its
USERPROFILE binding:

```rust
#[cfg(not(target_os = "windows"))]
#[test]
fn discover_sessions_excludes_rollback_backed_entries() {
    let _env_lock = lock_env();
    let tmp = tempfile::tempdir().unwrap();
    let home = tmp.path().to_string_lossy().to_string();
    let _env = EnvVarGuard::set_all(&[("HOME", &home)]);
    ...
```

### WR-04: Integration-test `ScopedEnvVar` duplicates `EnvVarGuard` without process-wide env lock

**File:** `crates/nono-cli/tests/audit_attestation.rs:261-283`
**Issue:** The new `ScopedEnvVar` RAII guard re-implements the
`crate::test_env::EnvVarGuard` shape (capture original via
`std::env::var().ok()`, restore on Drop). Two concerns:

1. **Missing env lock**: the in-tree `EnvVarGuard` users uniformly acquire
   `crate::test_env::ENV_LOCK` (or the equivalent `lock_env()`) before
   mutating env vars, because `std::env::set_var` is not thread-safe and
   Rust's test harness runs tests on multiple threads within the same
   process. Integration tests are no exception — they share the parent
   test process's env. `ScopedEnvVar` does NOT acquire any lock. The
   per-invocation env-var name (`NONO_TEST_AUDIT_KEY_VERIFY_<pid>_<nanos>`)
   avoids name collisions but does NOT protect the `set_var`/`var` calls
   from concurrent racing with other tests' env mutations (e.g., a future
   integration test that mutates `HOME`).

2. **Duplication risk**: when `EnvVarGuard` evolves (e.g., to add a
   poisoning check, or to use the new `unsafe { env::set_var }` shape that
   Rust 2024 will require), `ScopedEnvVar` will silently drift. The
   header comment justifies the duplication ("guard with `&'static str`
   key does not fit dynamically-named env vars") — accurate, but a
   one-line generic-over-`AsRef<OsStr>` extension to the in-tree guard
   would let both call sites share a single implementation.

**Fix (minimum):** Acquire `ENV_LOCK` before mutating. The in-tree lock is
crate-private but the symbol can be re-exported under a `#[cfg(test)]`
module path that integration tests can import. Or use a process-wide named
mutex (rare, fine for tests).

**Fix (preferred):** Generalize `EnvVarGuard` to accept `key: impl
Into<String>` and a single-key `set` constructor; then call it from both
unit tests and the integration test. Eliminates 25 lines of duplicated
code.

### WR-05: Deprecation-shim `warn_once_legacy_bundle_path` is unreachable for the common combo-session shape

**File:** `crates/nono-cli/src/audit_session.rs:158-176, 212-214`
**Issue:** The `warn_once_legacy_bundle_path` helper fires only when
`load_session` resolves a session via the rollback root (i.e.,
`!is_primary`). Because of BL-01, the audit-first iteration `?`-fails
before it can fall through to the rollback root for combo sessions —
which is exactly the shape the deprecation shim was designed to warn
about. As written, the shim only fires for sessions that exist solely
under the legacy rollback root with no audit-side directory created
(i.e., pre-Phase-22-05a sessions migrated forward, which is a small
historical subset).

This is a downstream symptom of BL-01 — fixing BL-01 (continue-on-metadata-failure
or mirror session.json) makes the shim reachable again.

**Fix:** Verify after BL-01 lands that the integration test exercises the
fallback path. If it does not, add a small unit test that constructs a
legacy-only session (rollback root has session.json, audit root does not)
and asserts `load_session` finds it AND `WARNED.get()` is `Some(())` after
the call.

### WR-06: `read_capability_decisions_from_ledger` reads from session.dir but combo-session ledger lives at audit dir

**File:** `crates/nono-cli/src/audit_commands.rs:438, 590, 321-349`
**Issue:** `cmd_show` calls `read_capability_decisions_from_ledger(&session.dir)`
to surface capability_decision counts. The `session.dir` returned by
`audit_session::load_session` for a combo session — once BL-01 is fixed —
will be EITHER the audit dir (where the ledger lives) OR the rollback dir
(where session.json lives), depending on which fix path BL-01 takes:

- If BL-01's fix is "mirror session.json to audit dir", `session.dir` will
  be the audit dir and the ledger read will succeed.
- If BL-01's fix is "continue on metadata-load failure", the loader will
  return the rollback dir for combo sessions, and the ledger read will
  silently return `Ok(vec![])` (the `File::open` Err branch on line 327)
  — capability decisions will be invisible in `audit show`.

The "graceful degradation" comment at lines 314-320 hides this from the
user: `audit show` will report 0 capability decisions when there are
actually N, with no surface signal.

**Fix:** Couple the fix decision for BL-01 with this. If the
"continue-on-failure" route is chosen, `audit_session::SessionInfo`
should carry an explicit `audit_dir: PathBuf` field separate from `dir`
(snapshot dir), and `read_capability_decisions_from_ledger` should be
called with the audit dir. If the "mirror session.json" route is chosen,
no change is needed here — but add an assertion that the ledger and
session.json are co-located.
