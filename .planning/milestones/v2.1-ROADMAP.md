# Milestone v2.1: Resource Limits, Extended IPC, Attach-Streaming & Cleanup

**Status:** ✅ SHIPPED 2026-04-21
**Phases:** 16 – 21 (inclusive; Phase 18.1 is a decimal gap-closure phase)
**Total Plans:** 25 plans across 7 phases
**Branch:** `windows-squash` (continuing from v2.0 + Phase 15)

## Overview

Deliver Job Object resource limits (CPU / memory / timeout / process-count), extend the Phase 11 capability pipe to broker additional Windows handle types end-to-end, finish the Phase 15 attach-streaming gap with a live `nono attach` flow on detached Windows sessions, re-establish Unix/macOS parity with upstream v0.37.1 (including a `rustls-webpki` security upgrade), land Windows single-file filesystem grants so the `claude-code` profile can actually run on Windows, and clean up the fmt/test/WIP debt accumulated during v2.0. Delivered against 13 v2.1 requirements (RESL, AIPC, ATCH, CLEAN, UPST, WSFG).

**Core value:** Windows security must be as structurally impossible and feature-complete as Unix platforms, ensuring the dangerous bits are kernel-enforced without compromising the supervisor-led security model.

## Milestone Summary

### Decimal Phases

- Phase 18.1: Extended IPC Gap Closure (inserted after Phase 18 for 5-gap HUMAN-UAT remediation post-21 end-to-end exposure)

### Key Decisions

- **Ship v2.0 with detached-console-grandchild bug as known issue, close via Phase 15:** Three fix directions in Phase 14 Plan 14-01 all failed the smoke gate; real fix required PTY + detached-supervisor architecture work which is its own investigation phase. Phase 15 direction-b (gated PTY-disable + null-token + AppID WFP) shipped 2026-04-18 and closed the carry-forward.
- **Direction-b scoped waivers for detached Windows path (Phase 15):** Low-IL isolation waived on detached path (Job Object + filesystem sandbox remain primary); per-session-SID WFP replaced by AppID WFP on detached path. Waivers scoped strictly detached-only.
- **Phase 17 ATCH-01 acceptance #3 downgrade (D-07):** Resize propagation on detached path is structurally infeasible because anonymous-pipe stdio (D-01, preserves Phase 15 `0xC0000142` fix) precludes ConPTY allocation. Users needing full TUI fidelity use `nono shell` or non-detached `nono run`.
- **AIPC `HandleKind` discriminators 0..=5 PINNED:** File=0, Socket=1, Pipe=2, JobObject=3, Event=4, Mutex=5 — wire-format stability lock for future cross-platform handle brokering extensions.
- **AIPC access-mask MAP DOWN, not DUPLICATE_SAME_ACCESS:** `broker_event/mutex/pipe/job_object_to_process` pass `dwOptions=0` + explicit mask so child handle is the validated subset, not supervisor source's full ALL_ACCESS (T-18-01-11 mitigation).
- **AIPC broker-failure flip (G-04):** Dispatcher flow-control rewritten so `(ApprovalDecision::Approved, grant=None)` is structurally unreachable via single-site `(decision, grant)` tuple construction + Err-arm flip to `Denied`. Wire-protocol compile-time tightening (`Approved(ResourceGrant)`) deferred to v2.2 to avoid cascading into 23 pre-existing tests.
- **AIPC privileged-port unconditional deny:** Socket broker enforces `port <= PRIVILEGED_PORT_MAX = 1023` reject BEFORE any profile-widening check; cannot be widened by `capabilities.aipc` profile (CONTEXT.md D-05 footnote).
- **WRITE_RESTRICTED capability pipe requires logon-SID co-requirement ACE (empirical discovery; MSDN undocumented):** Windows 11 26200's second-pass DACL access check for a `WRITE_RESTRICTED` token requires the DACL to contain TWO ACEs simultaneously — one matching a SID in `TokenRestrictedSids` AND one matching a group SID in `TokenGroups` with `SE_GROUP_MANDATORY`. `OW` Owner Rights does NOT satisfy the second co-requirement. Fix lands in `build_capability_pipe_sddl` via `current_logon_sid()` helper (commit `938887f`). Discovered via 13 SDDL variants tested through the new `crates/nono-cli/examples/pipe-repro.rs` harness.
- **CLEAN-02 hypothesis D-07 contradicted:** Parallel env-var contamination was NOT the root cause. All 5 Windows test flakes were 4 distinct deterministic Windows platform bugs — including a genuine production bug in `query_path` (UNC-prefix mismatch) that was user-approved as option C scope expansion.
- **CLEAN-04 auto-sweep threshold = 100 stale files + `NONO_CAP_FILE` structural no-op:** Prevents sandboxed agent from triggering host-side session-file deletion; `--older-than` breaking change from integer-days to require-suffix duration form rejects silent `30 seconds` vs `30 days` ambiguity.
- **Phase 20 `--allow-gpu` capability-routing deviation from upstream:** Fork routes directly through `CapabilitySet` + sandbox backend layer rather than upstream's `sandbox_prepare.rs::maybe_enable_macos_gpu`/`maybe_enable_gpu` path (known-risky given fork's 452-line vs upstream 1585-line divergence). `validate_oidc_issuer` fail-closed URL-component equality validator guards against `iss.starts_with` prefix-match anti-pattern.
- **Phase 21 Low-IL ownership pre-check (Rule-1+Rule-3 inline fix, commit `da25619`):** `try_set_mandatory_label` now skips `SetNamedSecurityInfoW` on paths not owned by the current user. Low-IL integrity is subtractive; Medium-IL system paths (e.g. `C:\Windows`) are already readable to Low-IL subjects through OS ACLs, so labeling them was never necessary — it was only ever effective on user-owned files.

### Issues Resolved

- Detached-console-grandchild `0xC0000142 STATUS_DLL_INIT_FAILED` carry-forward from v2.0 closed by Phase 15 direction-b fix (pre-v2.1 but folded in here for narrative continuity).
- 4 Phase 13 UAT items waived as `v2.0-known-issue` (P05-HV-1, P07-HV-3, P11-HV-1, P11-HV-3) promoted to `pass` by Phase 15 direction-b fix.
- Supervisor control pipe `ERROR_ACCESS_DENIED` (HRESULT 0x5) regression surfaced during Phase 21 end-to-end UAT: root-caused as undocumented `WRITE_RESTRICTED` + logon-SID co-requirement on Windows 11 26200; fixed in `938887f` + `e4c1bfa`.
- `WSAStartup` missing from Plan 18-04 child SDK `reconstruct_socket_from_blob` — masked in unit tests because test setup initialized WinSock via `WSADuplicateSocketW`; fixed in `e4c1bfa` via `OnceLock`-guarded `WSAStartup(MAKEWORD(2,2))` in an `ensure_winsock_initialized` helper.
- 3 latent pre-Phase-17 Windows session-id mismatch bugs (corrupted job-name format string in `create_process_containment` + 2 `self.session_id` vs `self.user_session_id` typos) fixed in drive-by commit `7db6595`.
- Phase 19 CLEAN-02 `query_path` UNC-prefix mismatch — a production bug where `nono why ~/.ssh` under-reported sensitive-path denials for non-existent paths on Windows because canonicalized paths carry the `\\?\` prefix while policy-expanded paths don't. Fixed via a local helper + `#[cfg(windows)]` regression test.

### Technical Debt Incurred

- **Phase 18 + 18.1 WR-01 stage asymmetry:** Pipe/Socket reject AFTER prompt due to per-kind validation ordering (direction/role/host/port checks happen after approval backend call). Event/Mutex/JobObject reject BEFORE prompt. Locked by tests; unification deferred to v2.2 as product decision.
- **Plan 18-03 `AipcResolvedAllowlist::default()` seed:** Resolved by Plan 18.1-03 Task 1 refactor — no longer outstanding.
- **Phase 20 non-security versions 0.32/0.33/0.34/0.35/0.36 upstream commits back-ported via manual port rather than cherry-pick:** Fork has ~452 LOC vs upstream 1585 LOC in `sandbox_prepare.rs`; cherry-pick conflicts would dominate plan time. Manual port preserved commit provenance via `Upstream-commit:` trailer.
- **Phase 21 `apply()` label-apply loop is best-effort per-file on system-owned paths:** Ownership-skip fix (commit `da25619`) silently no-ops on system paths. Correct on the merits (Low-IL is subtractive) but future readers will need the inline comment to understand why.
- **`NONO_CAP_FILE` env-var-based sandbox detection:** Phase 19 CLEAN-04 uses the same detection pattern as `reject_if_sandboxed` for the auto-sweep no-op. Works, but a more principled capability-layer detection is a v2.2+ candidate.
