# WIN-1706 Decision Package

`WIN-1706` is blocked on a product/support-contract decision, not on packaging or signing.

Already complete on this branch:

- release automation stages signed Windows `.exe`, machine MSI, user MSI, and zip artifacts
- Authenticode verification is wired into the release path
- MSI contract validation is automated
- docs/help/setup wording now describes the active Windows command surface accurately

Still blocked:

- [windows.rs](/cli/development/windows-security-model) support reporting still describes Windows as `partial`
- the library-wide `Sandbox::apply()` path is still not the backend path the CLI uses for supported Windows execution

That leaves two honest paths for maintainers.

## Option 1: Library/Runtime Alignment

Definition:

- Windows promotion requires the core library support contract to match the supported CLI/runtime behavior.

Result:

- Windows can only be called first-class supported after `Sandbox::support_info()` and `Sandbox::apply()` tell the same story as the CLI.

Code change summary:

- promote the core Windows sandbox implementation from a CLI-only/live-run path to a real library-supported path
- replace the current library `UnsupportedPlatform` apply behavior with support-aligned Windows execution
- move support reporting, runtime validation, and tests so they all describe one Windows contract instead of separate library and CLI stories

Design impact:

- best long-term architectural clarity: one Windows support contract at the library, CLI, docs, and release layers
- fewer future support ambiguities for embedders and maintainers
- higher implementation cost now, because backend/library alignment must be real before promotion is honest

## Option 2: CLI Surface Is The Product

Definition:

- Windows promotion is defined by the supported CLI command surface, even if the library remains partial.

Result:

- Windows can be called first-class supported at the CLI/release level while the library remains intentionally narrower.

Code change summary:

- keep the current Windows CLI/runtime behavior as the promoted product boundary
- leave the library support boundary narrower and explicitly partial
- split support messaging so setup/help/release/docs can describe CLI support without reusing the library support label as the whole product truth

Design impact:

- fastest path to product-level Windows promotion on the supported command surface
- preserves the current implementation shape with less backend churn
- creates an ongoing two-layer support model: CLI supported, library partial
- requires continuous documentation and review discipline so those two claims do not drift or get conflated

Primary code touch points:

- [windows.rs](C:/Users/omack/Nono/crates/nono/src/sandbox/windows.rs)
  - keep library support reporting intentionally partial, but document the narrower contract explicitly
- [setup.rs](C:/Users/omack/Nono/crates/nono-cli/src/setup.rs)
  - stop presenting library support status as the overall product support status
- [main.rs](C:/Users/omack/Nono/crates/nono-cli/src/main.rs)
  - keep the runtime contract framed as a supported Windows command surface rather than a library-wide support claim
- potentially add an explicit CLI-vs-library support distinction in CLI diagnostics instead of reusing the library status directly

Primary docs touch points:

## Recommendation

If maintainers want the cleanest long-term support story, choose **Option 1**.

If maintainers want to optimize for product readiness on the supported Windows CLI surface and are comfortable documenting a narrower library contract, choose **Option 2**.

What should not happen:

- claiming first-class Windows support in README/help/release messaging while neither option has been explicitly adopted
- changing `support_info()` wording alone without deciding whether the product bar is library-wide or CLI-surface-specific
