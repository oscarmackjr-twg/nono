# Windows Full Port Roadmap

This roadmap starts from the current repository state:

- Windows build, setup, dry-run, and direct execution already exist
- Windows supervisor IPC scaffolding exists
- Windows packaging exists as zip and MSI
- Windows still blocks unsupported enforcement-dependent flows intentionally

This plan covers the **full Windows port**, not just the current restricted-execution surface.

## Planning Constraints

The roadmap below is optimized for:

- **one engineer**
- **functionally equivalent Windows security** as the success bar
- **network enforcement included in the first full-port release**

That means this plan does **not** assume exact Landlock or Seatbelt semantic parity. It does assume that the Windows release must provide a defensible, enforceable Windows-native security boundary for the same core product promises:

- restricted process execution
- restricted filesystem access
- restricted network access
- supervised approval flow where supported
- rollback, state, and credential/trust handling that are production-usable

## Program Goals

The full port is complete when Windows is a first-class supported platform in:

- code
- CLI behavior
- test automation
- packaging and release
- operator documentation

That requires more than compiling and running commands. The Windows path must support real enforcement and must not rely on validation-only guardrails for core restricted workflows.

## Epic Overview

The full port is organized into six epics:

1. `WIN-EPIC-7` Windows Sandbox Backend
2. `WIN-EPIC-8` Supervisor Parity and Execution Hardening
3. `WIN-EPIC-9` Filesystem, Rollback, State, and Keystore Parity
4. `WIN-EPIC-10` CLI Feature Parity and Windows UX Completion
5. `WIN-EPIC-11` Windows Test Automation and Reliability
6. `WIN-EPIC-12` Packaging, Signing, and Release Readiness

The sequencing is intentional:

- backend first
- supervisor second
- parity work third
- automation and release hardening last

## Epic WIN-EPIC-7: Windows Sandbox Backend

Goal: replace preview classification and intentional blocking with real Windows enforcement.

### Stories

#### `WIN-701: Finalize Windows security model`
Points: 5

Acceptance criteria:

- chosen Windows enforcement primitives are documented
- capability mapping is explicit for filesystem, network, process, IPC, and child lifecycle controls
- unsupported semantics are identified before implementation starts
- the meaning of "functionally equivalent Windows security" is written down

#### `WIN-702: Implement process containment primitives`
Points: 8

Acceptance criteria:

- Windows child processes launch inside the chosen containment model
- parent/child lifetime rules are enforceable
- orphaned-process behavior is deterministic
- tests cover process lifecycle and containment failure paths

#### `WIN-703: Implement filesystem restriction backend`
Points: 13

Acceptance criteria:

- Windows enforces a supported subset of allowlist path restrictions
- denied paths fail deterministically
- protected paths remain blocked
- capability-to-policy mapping is documented and tested

#### `WIN-704: Implement network enforcement backend`
Points: 13

Acceptance criteria:

- live network restriction requests are truly enforced on Windows
- allow-all vs restricted network modes are distinct in runtime behavior
- blocked connections fail intentionally
- the implementation is documented and testable

#### `WIN-705: Return deterministic unsupported results for non-mapped capabilities`
Points: 5

Acceptance criteria:

- each capability class reports supported, blocked, or deferred intentionally
- no silent advisory-only success remains in live restricted execution
- CLI messaging reflects backend reality

#### `WIN-706: Wire enforced backend into live run path`
Points: 8

Acceptance criteria:

- live restricted runs use the real Windows backend
- supported restrictions execute successfully
- unsupported restrictions fail clearly
- call paths are test-covered

## Epic WIN-EPIC-8: Supervisor Parity and Execution Hardening

Goal: replace the current Windows supervisor scaffold with a real supervised runtime.

### Stories

#### `WIN-801: Implement Windows supervisor event loop`
Points: 13

Acceptance criteria:

- the parent event loop runs on Windows
- startup and shutdown are stable
- supervisor diagnostics are actionable

#### `WIN-802: Replace Unix fd-passing assumptions with Windows handle brokering`
Points: 13

Acceptance criteria:

- approved resources can be shared using Windows-safe handle transfer
- no Unix socket or SCM_RIGHTS dependency remains in supervised Windows flows
- transfer failure modes are testable

#### `WIN-803: Implement runtime capability expansion on Windows`
Points: 13

Acceptance criteria:

- supervised requests can expand access within the Windows security model
- denied requests fail safely
- audit and diagnostics capture the decision path

#### `WIN-804: Harden supervised teardown and recovery`
Points: 5

Acceptance criteria:

- interrupted sessions clean up correctly
- failed supervisor startup does not leave ambiguous state
- rollback/audit hooks behave consistently on supervisor failure

#### `WIN-805: Enable real supervised Windows run mode`
Points: 8

Acceptance criteria:

- supervised Windows mode is no longer scaffold-only
- CLI output no longer reports preview limitation for active supervised features
- process-level Windows tests cover the real supervised path

## Epic WIN-EPIC-9: Filesystem, Rollback, State, and Keystore Parity

Goal: make Windows storage, path, rollback, and trust behavior production-usable.

### Stories

#### `WIN-901: Harden Windows path normalization and protected-path semantics`
Points: 8

Acceptance criteria:

- canonicalization is Windows-safe
- drive-letter, UNC, junction, and symlink behavior are handled intentionally
- protected-path checks remain fail-closed

#### `WIN-902: Port rollback snapshot semantics to Windows`
Points: 13

Acceptance criteria:

- rollback snapshots work on Windows workspaces
- exclusion rules and large-directory behavior are documented
- Windows file-lock behavior is handled intentionally

#### `WIN-903: Validate state, audit, and runtime storage behavior on Windows`
Points: 5

Acceptance criteria:

- state roots resolve consistently for admin and user installs
- audit and runtime state do not depend on Unix conventions
- state migration or compatibility behavior is documented

#### `WIN-904: Keystore parity for credential and trust flows`
Points: 8

Acceptance criteria:

- Windows keystore-backed flows work for supported trust and credential paths
- missing or inaccessible secret-store behavior fails clearly
- tests cover expected Windows secret-loading scenarios

#### `WIN-905: Document Windows-specific filesystem and rollback semantics`
Points: 3

Acceptance criteria:

- docs explain Windows-specific path and rollback behavior
- operator guidance is aligned with actual implementation limits

- do not spend heavily on full CLI parity before the backend and supervisor model are real

## High-Risk Areas

The highest-risk items in this plan are:

- filesystem enforcement parity
- network enforcement in the first full release
- runtime capability expansion semantics on Windows
- rollback behavior under Windows file-locking and metadata rules
- Unix-to-Windows conversion of integration and regression coverage

## Definition of Done

Windows should only be called fully supported when:

- live restricted execution is truly enforced on Windows
- network restriction is truly enforced on Windows
- supervised mode is real and stable
- rollback, state, and keystore flows are validated
- main CLI workflows work on Windows, or permanent product differences are documented explicitly
- Windows tests run automatically in CI
- installers and binaries are signed for release use
- docs and runtime output no longer describe Windows as preview

## Immediate Next Story

The correct next story for the full port is:

- `WIN-701: Finalize Windows security model`

That story gates nearly every expensive implementation choice that follows, especially:

- process containment
- filesystem policy enforcement
- network restriction design
- supervisor-approved resource brokering
